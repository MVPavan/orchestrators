Agent SDK stack around LiteLLM Proxy

Consolidated sub-agent research on OpenAI Agents SDK, Anthropic/Claude tooling, Pydantic AI, LangGraph, the langchain-ai/deepagents repository, and LiteLLM Proxy after the FastAPI and React/Vite decisions.

LiteLLM stays gateway Use it for model routing, keys, spend, fallback, and gateway logging, not agent state.
Pydantic AI default Use as the typed Python agent SDK layer inside FastAPI.
App model is central Run, Session, ToolCall, Artifact, and Approval stay app-owned.
Sidecar SDK lanes Use OpenAI, Claude, LangGraph, or Deep Agents only where their native semantics matter.
01

Answer

Recommendation: make Pydantic AI the default in-process agent SDK layer behind FastAPI, talking to LiteLLM Proxy for model access. Do not make Pydantic AI the central product abstraction.

Central product abstraction: own the orchestration domain in application code and Postgres: Run, Session, Message, ToolCall, Approval, Artifact, Usage, TraceRef, and WorkflowState. SDKs are execution adapters under that model.

Use a thin custom LiteLLM path for one-shot/simple calls. Treat OpenAI Agents SDK as a credible conditional sidecar because it has official beta LiteLLM support, but require provider contract tests before making it central. Use Claude Agent SDK or Managed Agents as an adjacent Claude-native execution lane, not as a generic LiteLLM-backed client. Use LangGraph selectively after the worker/workflow decision if durable graph orchestration belongs inside the agent layer. Use Deep Agents as an optional LangChain worker harness for long-horizon research/code tasks, not as the default SDK.

Key Findings

Converging Evidence

  • LiteLLM is a good gateway, but not a universal feature-parity layer. Tool calling, strict structured outputs, Responses features, and provider-specific metadata need contract tests.
  • Pydantic AI maps best to a Python/FastAPI/Pydantic backend: typed dependencies, structured outputs, tools/toolsets, deferred approvals, streaming, evals, LiteLLM/OpenAI-compatible support, and official durable execution integrations.
  • OpenAI Agents SDK officially supports LiteLLM through the optional openai-agents[litellm] extra, LitellmModel, LitellmProvider, and litellm/... model names.
  • Pydantic AI's Temporal integration wraps agents with TemporalAgent and offloads model, I/O tool, and MCP communication to Temporal activities. OpenAI Agents SDK also has a Temporal integration path through temporalio.contrib.openai_agents.
  • SDK session/message history is not a product database. Durable sessions, artifacts, audit, tenancy, costs, and approvals need app-owned storage.
  • LangGraph is a serious runtime for interrupts/checkpoints/stateful agents, but choosing it centrally would preempt the worker/workflow decision.
  • Deep Agents is a standalone deepagents package from LangChain, but not an independent runtime: it is an opinionated harness on LangChain create_agent and LangGraph, bundling planning, filesystem/backends, subagents, context compression, memory/skills, HITL, and streaming.

Important Gaps

  • Remaining OpenAI Agents SDK risk is not LiteLLM absence; it is beta/best-effort adapter status and provider-specific gaps in tools, structured outputs, usage, tracing export, and Responses-only features.
  • Pydantic AI tools can return multimodal objects, but no researched source established a full artifact lifecycle with ACLs, storage, retention, previews, and provenance.
  • Temporal integrations do not remove app-level responsibility for product state, streaming projection, activity idempotency, payload sizing, approval audit, and reconciliation.
  • LiteLLM JSON mode is documented; strict schema parity across providers was not established.
  • No primary source showed first-class Deep Agents support for LiteLLM Proxy. Treat LiteLLM compatibility as indirect through LangChain chat-model integration and verify tool calling, structured output, streaming, and provider metadata with contract tests.
  • Worker durability, retry/resume, queueing, scheduling, cancellation, stream replay, and fanout belong to orch-0sc.4.

Failure Modes

RiskSeverityMitigation
SDK internals become the product's durable session/artifact database. High Persist canonical product state in app-owned Postgres tables and map SDK messages/events at boundaries.
LiteLLM silently drops or translates provider-specific behavior. High Disable silent parameter dropping for critical runs and run model-alias contract tests for tools, streams, schemas, and fallbacks.
Duplicate retry/fallback layers inflate cost and obscure failures. Medium Let LiteLLM own gateway retries/fallbacks; keep SDK retries scoped to tool-level recoverable errors.
OpenAI/Claude native sidecars split state and audit trails. Medium Wrap every native lane behind the same AgentRunService run/event/artifact contract.
LangGraph or Deep Agents absorbs workflow engine responsibilities prematurely. Medium Wait for orch-0sc.4; if used now, use only for a spike with explicit boundaries.
Sandbox/file/bash tools execute without enough isolation. High Require sandboxing, path allowlists, approval gates, audit logs, and artifact provenance before enabling destructive tools.

Validation Plan

Before implementation locks in package versions, run the same LiteLLM model alias through candidate execution paths. This belongs in implementation or spike work, not in this research artifact.

TestRequired EvidenceWhy It Matters
Non-streaming and streaming run Same alias works through Pydantic AI and thin LiteLLM path. Proves basic gateway contract.
Tool call round trip Model emits tool call, app executes, model consumes result. Agent SDK is useless without reliable tools.
Strict structured output Pydantic model validates across chosen providers. LiteLLM/provider parity is not guaranteed.
Fallback and budget exhaustion Clear error/event emitted and no duplicate runaway retry. Cost and reliability controls must be visible to UI/operators.
Correlation metadata Run ID appears in FastAPI logs, LiteLLM logs, SDK traces, and UI stream. Prevents observability fragmentation.
Human approval pause Tool request persists, UI approves/rejects, run resumes idempotently. Connects SDK behavior to product approvals.
