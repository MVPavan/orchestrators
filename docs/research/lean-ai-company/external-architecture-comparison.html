01

Answer

Yes, we can do the comparison. The useful conclusion is not "choose Paperclip" or "choose Gascity." The right move is to keep the accepted lean architecture as the spine, then selectively borrow external patterns that can be translated into app-owned Postgres state, Temporal workflows, and bounded adapter lanes.

Important correction: I found no formal repo artifact that defines a separate "just-in-time" or "JIT" architecture. In this report, JIT means an inferred implementation pattern: assemble the run snapshot, toolset, model alias, approval policy, workspace lease, and adapter config at run start, then freeze those choices for execution.

Most Paperclip qualities are implementable in the lean architecture, but not by copying Paperclip's Node service, Drizzle schema, SQL heartbeat queue, or plugin platform as-is. Most OpenHands qualities are implementable as product and runtime-boundary patterns, but OpenHands should not become the core execution substrate. Gascity and Gastown are strongest as references for reconciliation safety and coding-lane mechanics, not as replacements for Temporal and app-owned Postgres.

I interpret the user's "R architecture" phrasing as "our architecture": the accepted lean/minimal architecture documented in final-minimal-architecture.html.

Pros And Cons

Option Pros Cons How To Use It Do Not Use It When
Lean minimal Clear truth ownership; good fit for FastAPI decision; Temporal covers retries, timers, signals, cancellation; Postgres supports audit and UI projection; CLI coding lanes can preserve subscription login. Requires explicit schemas, migrations, reconciliation, auth, tool registry, artifact service, contract tests, and worker discipline. It does not give Paperclip/OpenHands breadth for free. Keep as the base architecture. Borrow external features only after mapping them to Postgres domain state and Temporal workflows or activities. You need immediate full plugin marketplace, multi-tenant SaaS governance, and broad adapter parity before proving one vertical slice.
JIT run assembly Keeps product flexible while retaining replayability: each run freezes active spec, role, model alias, tool versions, approvals, context, workspace, and adapter mode. Can become chaos if it means mutable live config, framework-owned session state, or ad hoc prompt/tool mutation during execution. Use as an implementation technique inside lean minimal: resolve late, snapshot once, execute deterministically. You want arbitrary live edits to long-running workflows without migration/version rules.
Gascity Excellent desired-state safety model; typed domain core with CLI/API projections; provider/store abstractions; order triggers; partial-query safeguards to avoid destructive reconciliation. Custom control loop complexity; Beads/session state becomes central; many provider modes; worker boundary still has migration complexity. Borrow reconciliation patterns: partial-read flags, provider health snapshots, idempotency/epoch fences, conservative drains, typed projection APIs. You are tempted to replace Temporal with a custom supervisor tick loop or put product truth in Beads.
Gastown Strongest local coding-lane reference: isolated worktrees, named worker identities, adapter presets, git evidence, merge/refinery gates, recovery daemon. Distributed truth across Beads, Dolt, git, tmux, files, and CLI side effects; Dolt dependency; provider flag drift risk. Borrow coding-run mechanics: worktree leases, branch naming, evidence capture, review/apply gates, provider compatibility metadata. You need a clean web product state model or multi-user product auth before local CLI orchestration breadth.
Paperclip Closest external model to an AI-company product: durable agents/issues/runs/costs, workspaces, environments, adapters, plugins, routines, secrets, budgets, UI/CLI/MCP API clients. Not lean; broad service layer; heartbeat path composes many domains; custom SQL queue/recovery overlaps with Temporal; plugin workers and plugin migrations expand the security surface. Borrow schema ideas and correctness features: wakeups/runs, issue locks, workspace-finalize barrier, environment leases, adapter auth checks, UI/CLI/MCP as API clients. You want to preserve Temporal as execution truth and a Python/FastAPI/Pydantic codebase without a full plugin platform.
OpenHands Strong FastAPI/React app-server precedent; useful async start-task UX, sandbox health model, session API key boundary, provider integrations, MCP PR/MR tools, enterprise overlay lessons. Execution semantics live behind pinned SDK/agent-server packages; no local Temporal/Pydantic AI usage found; direct runtime URLs and file-backed settings/secrets are not product-truth patterns for our system. Borrow app-server/runtime boundary ideas, start-task records, sandbox adapter model, event callbacks, provider abstractions, and tenant/auth lessons. You are considering making OpenHands SDK/agent-server the central runtime instead of Temporal plus Pydantic AI.

Recommendation

Keep The Base

Keep the accepted lean architecture: FastAPI control plane, React/Vite UI, Postgres product truth, Temporal execution truth, LiteLLM model gateway, Pydantic AI default SDK adapter, app-owned registries, local artifact claim-check storage, and CLI-first Codex/Claude coding lanes.

Adopt A Narrow Borrow List

Borrow only four external ideas in the first implementation pass: Paperclip's run/lock/workspace schema ideas, OpenHands start-task/sandbox UX, Gascity partial-evidence reconciliation safety, and Gastown worktree/evidence/merge-gate mechanics.

Concrete Next Architecture Tasks

Order Task Borrowed From Gate
1 Define minimal app domain schema: run, run_snapshot, message, approval, tool_call, artifact, usage, workspace_lease, adapter_invocation. Paperclip plus lean final architecture. Schema supports one generic run with audit and replay-safe IDs.
2 Define start-task and event projection API for UI. OpenHands start-task pattern. FastAPI returns start task immediately and streams Temporal-backed progress through SSE.
3 Define reconciliation and repair contract between Postgres, Temporal, artifact bytes, and workspace leases. Gascity partial-read safety. No destructive cleanup when evidence is partial; operator-visible repair path exists.
4 Define coding-lane adapter contract: auth mode, command, resume, output parser, worktree lease, cancellation, logs, diffs, tests, artifact capture. Gastown plus Paperclip adapter metadata. Codex and Claude CLI adapters can run in subscription mode without shared-checkout writes.
Red-team conclusion

The strongest argument against the recommendation is that Paperclip is already closer to a productized AI-company control plane than the lean architecture. That is true, but copying Paperclip would import a broad Node/Drizzle/service/plugin architecture and duplicate Temporal's chosen responsibilities. The lean stack is still the cleaner starting point because it makes the ownership boundary explicit before product breadth is added.
