Agent role and department specification model

Consolidated sub-agent research on representing departments, roles, workflows, prompts, model policy, tool grants, approvals, context scopes, and auditability for the FastAPI, Postgres, Temporal, Pydantic AI, LiteLLM, and app-owned tool-registry architecture.

Postgres specs Canonical role, department, workflow, prompt, model, tool, context, and approval versions.
Code primitives Temporal workflows, agent factories, validators, tools, adapters, and policy evaluator stay in code.
Run snapshots Every run pins immutable versions and hashes. No long run binds to latest config.
No policy engine yet Use typed app policy tables in v1. Defer OPA, Cedar, and OpenFGA until scale demands them.
01

Answer

Recommendation: build an app-owned Spec Registry in Postgres with immutable, publishable versions for departments, roles, workflows, prompts, model policy, tool grants, context scopes, approval policy, and adapter policy. Generate Pydantic AI agents/toolsets and Temporal run inputs from a per-run snapshot of those versions.

Do not make Pydantic AI AgentSpec, DeepAgents subagent configs, CrewAI crews, AutoGen teams, OpenAI Agents SDK agents, or mutable JSON blobs the source of truth. Treat them as adapter/import/export shapes.

The clean split is: policy semantics and execution mechanics in code, policy parameters and approved versions in Postgres. The database may store prompt text, ordered instruction blocks, model aliases from an allowlist, tool IDs and versions from the internal registry, context scope IDs, approval thresholds, cost ceilings, and lifecycle status. It must not store executable Python, import paths, shell snippets, SQL fragments, arbitrary predicates, provider credentials, or admin-authored mini-programs.

A department is not just a group chat. A role is not just a prompt. A workflow is not just a list of agents. These are separate, versioned contracts: departments define membership and coordination style; roles define authority and behavior; workflows define Temporal execution structure, checkpoints, handoff edges, and termination rules.

02

Scope And Method

Research was delegated to four sub-agents: Pydantic AI and selected SDK mechanics, Temporal workflow/versioning, multi-agent framework role patterns, and policy/governance. The orchestrator spot-checked primary docs for the most decision-sensitive claims and treated prior local decisions as fixed constraints.

Fixed DecisionConstraint For This Model
FastAPI and Postgres own product state. Specs, approvals, audit events, run snapshots, and UI-visible state are app tables and APIs.
Pydantic AI is the default in-process SDK. Generate Pydantic AI agents from app snapshots; do not store SDK object graphs as product truth.
Temporal owns durable workflow execution. Workflow code stays deterministic and stable. External I/O, DB reads, model calls, and tool calls happen in activities or SDK-managed activity paths.
App-owned Capability Registry owns tools. Role specs reference tool IDs, versions, grants, and approval policy; they do not self-authorize tools.
Subscription-first adapters where practical. Adapter policy must include auth_mode. Codex and Claude Code CLI lanes stay controlled coding-run tools, not generic shell access.

Recommended Architecture

Solid arrows show publish and run-start flow. Dashed arrows show runtime enforcement: Temporal and adapters consume pinned snapshots and call back into app-owned policy, tool, approval, and audit services.

LayerApp-Owned DataCode-Owned Behavior
Department department_spec_version, role membership, coordination pattern, shared scopes, escalation policy. Department router, run planner, UI validation, lifecycle transitions.
Role role_spec_version, router description, ordered instructions, model/tool/context/approval refs, output contract key. Pydantic AI agent factory, output validators, dependency builders, guardrail hooks.
Workflow workflow_spec_version, role bindings, task graph, handoff edges, checkpoint rules, termination rules. Temporal workflow classes, activity definitions, update/signal/query handlers, worker versioning.
Policy Model profiles, tool grants, context scopes, approval policies, adapter auth mode, cost/time limits. Small deterministic policy evaluator, secret lease checks, sandbox enforcement, registry contract tests.
Run agent_run, agent_run_snapshot, pinned version IDs, hashes, current Temporal Run ID, audit events. Run start orchestration, snapshot rendering, idempotent activity writes, reconciliation.

Minimum Spec Contract

ContractFields To StoreRules
department_spec_version department_key, version, status, purpose, owner, allowed roles, coordination pattern, default context boundaries, escalation route, content hash. Department config selects from allowed coordination patterns such as router, supervisor, sequential workflow, parallel review, or coding-run lane. It does not define executable orchestration code.
role_spec_version role_key, department, version, status, purpose, description_for_router, instruction block refs, model policy ref, tool grant set ref, context scope ref, output contract key, approval policy ref. Role is authority plus behavior. It must include tool, context, model, output, approval, and audit policy, not only prompt text.
instruction_block_version kind, content, variable schema, source, reviewer, hash, rendered prompt retention policy. Use ordered blocks for base, role, workflow, safety, and task instructions. Do not rely on provider-specific system-prompt semantics.
model_policy_version LiteLLM alias, allowed aliases, model settings, max tokens, budget, fallback policy, structured-output/tool-call contract-test status. Fallbacks are allowed only when contract tests prove equivalent behavior for tools and output contracts.
tool_grant_set_version Capability registry snapshot ID, tool IDs and versions, side-effect class, secret grant alias, sandbox/network policy, approval requirement. References the app-owned Capability Registry. No raw MCP catalog, generic shell, broad filesystem, or broad SQL access by default.
context_scope_version tenant/resource scopes, memory/RAG scopes, artifact classes, redaction policy, allowed deps fields, untrusted-content labeling policy. Context is fetched through scoped app APIs or activities. SDK memory/session helpers are not system-of-record memory.
workflow_spec_version Temporal workflow type, task queue, input schema, role bindings, handoff edges, approval checkpoints, termination rules, minimum worker deployment/build. Workflow specs bind product intent to code-owned Temporal primitives. Code changes use Temporal worker versioning or patching, not DB mutation.
adapter_policy_version adapter type, auth_mode, permission profile, environment allowlist hash, workspace isolation, CLI flags profile, subscription/API-key gate. Codex and Claude Code subscription-compatible CLI adapters are explicit coding tools. Role specs never store OAuth tokens, keys, or host env.

05

Framework Lessons

Source PatternAdoptAvoid
Pydantic AI AgentSpec Use as import/export shape for model, instructions, capabilities, settings, deps schema, output schema, retries, and metadata. Do not make it canonical. Its output_schema is an instruction to the model, not hard runtime validation of properties/required fields.
Pydantic AI agents/toolsets/deferred tools Generate agents from snapshots; map deferred tool requests and approval wrappers to app-owned approvals. Do not let SDK message history replace app-owned run/session/tool/audit tables.
Temporal durable workflows Use stable workflows, activities for I/O, Updates for accepted/rejected approval semantics, Queries for status, and Worker Versioning for code changes. Do not branch workflow replay on live DB latest config. Do not store PII/secrets in workflow IDs or search attributes.
DeepAgents, LangGraph, CrewAI, AutoGen Borrow vocabulary: subagents, supervisor/router patterns, handoff edges, scoped memory, teams, tasks, flows, termination rules. Do not adopt any of them as the central product authority for roles, departments, memory, or permissions in v1.
OpenAI Agents SDK Borrow handoff shape: target, description, optional input schema, history policy, hooks, and enablement. Do not make provider-native handoffs bypass app policy, tool grants, or run snapshots.
OPA, Cedar, OpenFGA Keep as upgrade options for cross-service policy, formal auth policy review, and relationship-heavy access control. Do not add a general policy engine before v1 has enough services, tenants, or delegated administration to justify it.
06

Lifecycle And Gates

StageWhat HappensGate
Draft Create or edit a new version row. Existing active versions remain immutable. Schema validation, references exist, no unsafe executable fields.
Validate Resolve model aliases, tool registry refs, context scopes, prompt variables, output contract keys, workflow bindings, and adapter auth mode. Contract tests for selected model/tool/output path; policy evaluator dry run; permission expansion diff.
Approve Human or privileged role approves the exact content hash and expansion diff. Approval binds actor, spec type, version, hash, timestamp, reason, and scope.
Activate Version becomes selectable for new runs. Existing runs stay pinned to their snapshot. No automatic migration of long-running Temporal executions.
Run FastAPI resolves exact versions, writes agent_run_snapshot, starts Temporal with non-PII workflow ID, and records Temporal Run ID chain. Idempotency key and unique constraints prevent duplicate starts/effects.
Deprecate Stop using a version for new runs while preserving it for replay, audit, and active runs. Replacement exists or explicit exception recorded.

Decision Matrix

OptionBest FitSacrificeVerdict
Hardcoded Python roles only Earliest prototype with two or three roles and no admin editing. No durable product configuration, weak audit of prompt/tool changes, slower iteration. Accept only for spike
Mutable DB JSON agent configs Fast UI editing with minimal schema work. Authority drift, replay drift, unsafe mini-language pressure, hard audits. Reject
Versioned app specs + code factories Lean v1 that needs flexible roles without giving the database arbitrary execution power. Requires explicit schema, publish workflow, and validation discipline. Choose
Pydantic AI AgentSpec as canonical DB Portable import/export and editable agent config files. SDK-shaped authority, capability allowlist risk, output-schema validation caveat. Adapter format only
External multi-agent framework owns roles Teams/crews/swarms where the framework is the whole product. Weak fit with app-owned audit, Temporal snapshots, subscription-first coding adapters, and internal tool registry. Reject for v1
OPA/Cedar/OpenFGA in core v1 Large cross-service policy, formal auth policy, or relationship-heavy sharing. More operational surface before authorization complexity exists. Defer
08

Failure Modes

RiskSeverityMitigation
Database-configurable behavior becomes arbitrary code through imports, Python snippets, shell, SQL, HTTP, templates, or free-form predicates. High Allow only typed references to code-owned primitives. Any future expression language is treated as code: reviewed, tested, versioned, and deployed.
Prompt or tool changes silently expand role authority. High Version prompt, model, tool, context, approval, and adapter policy together through snapshot hashes and expansion diffs.
Temporal workflow replays with different live config. High Resolve exact config once and pin it in agent_run_snapshot. Workflow code uses IDs/hashes and activities, not live latest rows.
Approval replay or scope leak. High Approval rows bind actor, role, policy version, tool ID/version, args hash, scope, expiry, and run ID. Changed args invalidate approval.
Model alias fallback changes structured output or tool calling semantics. Medium Require LiteLLM alias contract tests before fallback is allowed for a role/output/tool path.
Audit logs leak prompts, secrets, or user data, or become useless after over-redaction. Medium Redact payloads, store hashes and correlation IDs, separate secret grant IDs from secret values, and preserve artifact refs.
Manual DB edits bypass draft/validate/approve lifecycle. Medium Use append-only version rows, status transitions through app code, DB constraints, and admin audit events.
09

Implementation Decisions

QuestionDecisionFirst Spike
What is the first editable unit? role_spec_version with ordered instruction blocks, model policy, context scope, tool grant set, output contract key, and approval policy. Create one research role and one reviewer role; generate Pydantic AI agents from a snapshot.
How are departments represented? Department specs define role membership, coordination pattern, shared scopes, and workflow template refs. Create a coding department with planner, implementer, reviewer, and CLI coding-run tool grants.
How are handoffs represented? Explicit edges with source role, target role, description, payload schema, history filter, approval policy, and resume behavior. Spike planner-to-implementer and implementer-to-reviewer handoffs with private context by default.
Where do approvals live? App approval_checkpoint rows are authoritative. Temporal Updates are used when the caller needs accepted/rejected/completed semantics. Pause a tool grant expansion and approve it through FastAPI, then resume the workflow.
When should OPA/Cedar/OpenFGA be added? Not in v1. Add OPA for cross-service policy, Cedar for formal auth policy review, OpenFGA for relationship-heavy resource sharing. Keep a tiny app evaluator with typed inputs and tests; log every decision for possible future migration.
