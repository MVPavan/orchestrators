FastAPI as the core web app framework

Consolidated sub-agent research on whether a personal-assistant-led multi-agent company system can be built cleanly on FastAPI without Django, and what library set is needed to cover Django-like gaps.

Choose FastAPI As the HTTP/API and realtime-status edge, not as the durable workflow engine.
Medium-high Confidence, assuming external auth or a deliberately scoped auth module.
Auth/security The main gap versus Django, ahead of ORM or request routing.
No blocker But the missing batteries must be treated as architecture decisions, not defaults.
01

Answer

Recommendation: use FastAPI as the core web/API framework for the first version if the product is an API-first control plane with chat ingress, workflow/status APIs, approval endpoints, artifact metadata, LiteLLM integration, and SSE/WebSocket status streams.

Do not treat this as a vote for FastAPI as the whole application platform. FastAPI should own HTTP, typed contracts, OpenAPI, dependency wiring, streaming endpoints, and ASGI lifecycle. Postgres, workers/workflows, auth/session policy, admin/internal inspection, migrations, and observability need explicit components.

Django becomes the better default if the first release is mostly a browser-facing relational control plane: staff/admin CRUD, server-rendered forms, local users, groups, permissions, CSRF-protected cookie sessions, and a model-admin workflow. A hybrid Django-plus-FastAPI shape is possible, but it should be avoided initially unless the boundary is clean enough to justify two web frameworks.

03

Key Findings

Facts

  • FastAPI is API-first: Python type hints, Pydantic validation, Starlette ASGI foundation, OpenAPI/JSON Schema, dependency injection, WebSockets, StreamingResponse, SSE documentation, lifespan hooks, middleware, and TestClient support.
  • Django bundles a coherent application stack: auth, groups/permissions, sessions, CSRF, forms, ORM, migrations, admin, tests, management commands, and security guidance.
  • FastAPI security helpers wire schemes and OpenAPI semantics; they do not create a complete product auth/account/session system.
  • FastAPI BackgroundTasks and Starlette background tasks are after-response hooks, not durable workflow infrastructure.
  • SQLAlchemy plus Alembic is mature enough for the persistence layer, but Alembic autogenerate still requires migration review.
  • Django 5.2 is an LTS release with support listed through April 2028; FastAPI remains in 0.x and official docs tell users to pin versions for production.

Inferences

  • FastAPI fits a control-plane API better than Django when the UI is custom, auth can be delegated or narrowed, and streaming/status endpoints matter.
  • The main FastAPI cost is integration discipline around auth, sessions, CSRF, admin, and operational defaults.
  • Django's strongest value is reducing the number of choices in boring, security-sensitive, database-backed product surfaces.
  • Request-router performance is not the key decision variable for an agent orchestration product; LLM calls, queue latency, Postgres, artifact IO, and worker scheduling will dominate.

Minimal FastAPI Stack

The recommended FastAPI stack is intentionally small. It should be a pinned, documented baseline rather than a grab bag of plugins.

NeedRecommended ComponentReasonRisk
Web/API layer FastAPI + Starlette + Uvicorn Typed APIs, OpenAPI, ASGI, SSE/WebSocket support, testability. Pin versions
Data model and migrations SQLAlchemy 2.x + asyncpg + Alembic Mature Postgres persistence and migration tooling. Review migrations
Settings pydantic-settings Matches FastAPI/Pydantic type system and environment-driven config. Low
External identity Authlib + first-party user/role mapping Best path if OIDC/OAuth is acceptable; avoids local password lifecycle early. Policy work remains
Local account flows FastAPI Users only behind an internal auth boundary Useful for registration/login/reset/verify, but project docs say maintenance mode. Migration risk
Internal admin SQLAdmin, behind strong auth Narrow SQLAlchemy-aligned CRUD/inspection surface. Beta maturity
Sessions Server-side/session strategy tied to auth; Starlette SessionMiddleware only for low-sensitivity state Starlette signed-cookie sessions are readable by clients and not server-revocable by themselves. CSRF/session bugs
Background work None in this decision; defer durable work to the worker/workflow BD item FastAPI BackgroundTasks are not a durable job system. Do not overuse

Explicit exclusions for now: fastapi-sessions because the latest reported PyPI release is stale, SQLModel as the architecture anchor because SQLAlchemy is the mature substrate, standalone WTForms unless the UI decision later requires custom server-rendered forms, and publicly exposed admin packages because admin surfaces should stay behind strong internal auth.

07

Counterargument

The strongest case against FastAPI is not that it lacks routing, validation, or async support. It is that a solo or small-team, long-lived assistant product may spend most of its risk budget on account management, internal inspection, approvals, staff workflows, relational data repair, sessions, CSRF, permissions, and schema evolution. Django solves much of that in one coherent stack and has an LTS path.

The strongest FastAPI rebuttal is that this system is not a conventional CRUD/admin product if the architecture direction stays as stated: Python, FastAPI, Postgres, LiteLLM Proxy, local artifacts, Docker Compose, agent/tool services, streaming status, and later workers. In that shape, Django's bundled UI/admin stack is valuable but not central enough to justify making the API/control-plane layer less natural.

Decision FlipMeaning
Flip to Django First release needs local users, staff CRUD, groups/permissions, CSRF forms, Django-admin-like operations, and low-maintenance browser session security more than API streaming ergonomics.
Stay with FastAPI External auth is acceptable, APIs dominate, UI is custom, streaming status is central, and the team commits to a pinned library baseline plus security tests.
Consider hybrid Django and FastAPI can own separate products or services with a clean boundary. Avoid this if they would share the same auth/session/workflow surface.

Decision Points And Open Questions

Open QuestionWhy It MattersSuggested Default
Is external OIDC acceptable for the first deployment? Determines whether Authlib is enough or local password lifecycle is needed. Prefer external OIDC if available.
Will the UI use cookie-backed browser sessions? Cookie auth makes CSRF and server-side session revocation first-order requirements. Assume yes for browser UI and design CSRF now.
Is admin CRUD allowed to mutate production data? Changes risk posture for SQLAdmin or any internal inspection tool. Start inspection-first; curated repair tools later.
Which worker/workflow engine wins orch-0sc.4? FastAPI should not carry durable orchestration semantics. Keep FastAPI thin until that decision lands.
Which UI layer wins orch-0sc.2? Server-rendered forms versus SPA affects session/CSRF/admin choices. Defer final form/admin choices until UI research is reviewed.
