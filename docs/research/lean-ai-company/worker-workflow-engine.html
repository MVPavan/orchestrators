Worker queue and workflow engine architecture

Consolidated sub-agent research on Temporal, Prefect, Postgres plus Procrastinate, Redis-backed queues, and custom Postgres workflow state for the FastAPI/Pydantic AI/LiteLLM assistant system.

Choose Temporal Use it as the durable execution engine for human-gated agent runs.
Postgres stays truth Run, approvals, artifacts, audit, and UI events remain app-owned.
Prefect is sidecar Credible for AI/data/ops workflows, not the core product workflow engine.
No Redis state Use Redis only for cache/pubsub/rate-limit style work unless explicitly hardened.
01

Answer

Recommendation: choose Temporal as the durable workflow engine for core agent-run orchestration, with app-owned Postgres as the product and audit source of truth.

Do not choose Prefect, Redis-backed queues, or custom Postgres queue logic as the primary workflow engine. Do not pair Temporal and Prefect for the same v1 agent-run lifecycle.

The deciding factor is not queue throughput. This product needs restart-safe multi-step runs, approval pauses, cancellation, retries, fan-out/fan-in, and resumability across deploys or process loss. Temporal is built for those durable execution mechanics. Postgres should still own the business model: runs, sessions, approvals, artifacts, event log, costs, audit, and UI-visible state.

The small custom layer should be an adapter and mirror, not a workflow engine: Run.workflow_id maps to a Temporal workflow, ToolCall.activity_ref maps to activities, Approval rows drive Temporal signals or updates, and artifacts remain claim-check references in local storage.

Correction pass: Pydantic AI and OpenAI Agents SDK both have Temporal integration paths. That strengthens the Temporal choice rather than replacing it. Prefect can orchestrate AI workflows through Pydantic AI PrefectAgent, deployments, workers, events, and human input, but it remains a better sidecar for scheduled/evaluation/data/ops agent jobs than the default engine for user-facing durable agent runs.

Scope And Method

Research was split across sub-agents covering Temporal, Prefect, lightweight queues and custom state, and a red-team composition pass. A focused correction pass then validated Pydantic AI Temporal, OpenAI Agents SDK Temporal, and Prefect AI/agent orchestration claims. Source weighting favored official docs, current package docs, and implementation guidance. Practitioner posts were used only for operational reality and are labelled as weaker than primary docs.

Fixed DecisionConstraint For This Choice
FastAPI is the backend/API edge. FastAPI starts workflows, sends approvals/cancellations, exposes state and streams; it does not run durable workflows itself.
React + Vite is the UI. The browser consumes app-owned status/event APIs, not raw Temporal, Prefect, SDK, or queue internals.
Pydantic AI is the default agent SDK. Use TemporalAgent.run() inside workflows; its integration offloads model, I/O tool, and MCP communication to activities.
OpenAI Agents SDK is a conditional sidecar. Temporal's Python SDK exposes temporalio.contrib.openai_agents; use it if that lane is adopted, not a custom wrapper first.
LiteLLM Proxy is the model gateway. Gateway retries/fallbacks stay model-layer concerns; workflow retries must wrap idempotent activities.
App-owned domain model is central. Workflow engine state is execution state. Product truth and audit remain in Postgres.

05

Key Findings

Converging Evidence

  • Temporal workflows are durable, replayed from event history, and designed for long-running processes that can sleep without consuming worker compute.
  • Temporal signals, queries, and updates map cleanly to human approval and live status interactions, but the app Approval table should remain the audit authority.
  • Pydantic AI has an official Temporal durable-execution integration via TemporalAgent, PydanticAIPlugin, and related helpers. It wraps an agent for deterministic workflows and offloads model, I/O tool, and MCP communication to activities.
  • OpenAI Agents SDK also has an official Temporal path, primarily documented in Temporal's Python SDK as temporalio.contrib.openai_agents; OpenAI package metadata exposes a temporal extra.
  • Procrastinate is the strongest lightweight queue because it is Postgres-backed and can enqueue within an externally managed app DB transaction.
  • Prefect is credible for AI-assisted scheduled/data/ops workflows through Pydantic AI PrefectAgent, deployments, workers, events, automations, and human input. This is useful durable AI task execution, but not the same architectural primitive as Temporal's replayed workflow execution.
  • Redis queues are useful for job distribution, but Redis should not be the only record for critical workflow state unless it is deliberately operated as a durable HA database.

Important Boundaries

  • Temporal owns execution mechanics, not the product database.
  • Workflow code must stay deterministic; use supported SDK integrations such as Pydantic AI TemporalAgent or OpenAI Agents SDK OpenAIAgentsPlugin rather than plain provider clients inside workflows.
  • Every activity that writes Postgres, artifacts, or external tools must be idempotent against run_id, step_id, or a stable operation key.
  • Large prompts, transcripts, artifacts, and model outputs should not be stored directly in Temporal event history.
  • LangGraph and Deep Agents may manage agent reasoning state, but they should not become the infrastructure workflow engine.
  • "Prefect agents" is ambiguous in current Prefect 3 terminology. Legacy Prefect agents became workers; AI agents appear through Pydantic AI PrefectAgent, Prefect MCP, examples, or separate PrefectHQ projects such as Marvin.

Failure Modes

RiskSeverityMitigation
State authority splits between Postgres, Temporal, queue rows, and UI cache. High Define Postgres as product/audit truth and Temporal as execution truth. Project workflow events into app tables with explicit reconciliation.
Non-deterministic workflow code breaks replay. High Keep workflow functions deterministic. Use Pydantic AI TemporalAgent or OpenAI Agents SDK OpenAIAgentsPlugin so model/tool/MCP I/O is activity-backed.
Activity retries duplicate external side effects. High Use operation keys, idempotent writes, unique constraints, and explicit compensation where a tool/API cannot be idempotent.
Temporal event history grows with prompts, tokens, or artifacts. Medium Use claim-check references to app Postgres/local artifact storage. Continue-as-new for long histories.
Temporal self-hosting becomes the main operations burden. Medium Start with local dev server or Compose. For production, prefer Temporal Cloud unless self-hosting has an owner, backup plan, metrics, and upgrade path.
Temporal + Prefect creates duplicate workflow owners. Medium Do not make Prefect and Temporal own the same process. Add Prefect only for separate scheduled/eval/data/ops agent jobs with a clear boundary.
SDK integration limitations are hidden until implementation. Medium Spike Pydantic AI TemporalAgent first, then OpenAI Agents SDK Temporal only for a real sidecar need. Check streaming, context, sandbox, and unsupported tool/session features explicitly.

Implementation Decision Points

QuestionDecisionFirst Spike
What is the unit of orchestration? One Temporal workflow per app Run; child workflows for durable sub-agent or department-level work. Create a minimal RunWorkflow with one model activity, one approval wait, cancellation, and event projection.
Where does human approval live? App Postgres Approval row is authoritative. Temporal waits for a signal/update keyed to that row. FastAPI approval endpoint writes the row, then sends the Temporal message. Reconcile on retry.
Where do Pydantic AI and LiteLLM run? Use TemporalAgent.run() from workflow code; the integration offloads model requests, I/O tools, and MCP communication to activities. Run one named TemporalAgent through LiteLLM Proxy, with stable tool IDs, explicit activity retry policy, and an idempotent ToolCall write.
What about OpenAI Agents SDK? Use Temporal's temporalio.contrib.openai_agents integration only for a deliberate OpenAI-native sidecar lane. Spike one workflow using OpenAIAgentsPlugin, one model call activity, one activity_as_tool, and tracing/context propagation.
What about Prefect AI agents? Keep Prefect as a sidecar candidate for scheduled AI/data/ops workflows, evaluations, maintenance, and analyst jobs. Do not add it to the v1 core run path. Later spike Pydantic AI PrefectAgent for recurring eval/report jobs.
What if Temporal is too heavy for the first prototype? Use Postgres + Procrastinate only as an explicit deferral path, with migration triggers documented. Same app tables and service interface; swap execution adapter later without changing UI/API contracts.
What are the migration triggers? Temporal is mandatory once runs routinely wait on humans, span hours/days, require retry across deploys, need fan-out/fan-in, or custom state-machine code starts growing. Track these in implementation planning rather than discovering them after custom orchestration exists.
