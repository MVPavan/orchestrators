Local artifacts and Docker Compose baseline

Consolidated sub-agent research validating a lean local-storage and single-host Compose baseline for the FastAPI, React/Vite, Postgres, Temporal, Pydantic AI, LiteLLM, tool-registry, and subscription-first coding-adapter architecture.

Local files v1 Use filesystem bytes with Postgres metadata and authorization. No large artifact payloads in Temporal or Postgres blobs.
Object-shaped keys Design storage keys and metadata for S3 migration from day one, even while using local filesystem.
Compose baseline Postgres, init jobs, Temporal, LiteLLM, FastAPI, worker, proxy, static UI. MinIO is optional.
Backup as pair DB dump plus artifact archive share one backup set and restore drill. No live Postgres volume tar.
01

Answer

Recommendation: confirm local artifact storage plus Docker Compose as the v1 baseline. Implement artifacts as a claim-check store: Postgres owns metadata, access control, retention, and audit; the local filesystem owns immutable bytes; Temporal and agents carry only compact artifact references.

Do not add MinIO, S3, Kubernetes, managed Postgres, or Temporal Cloud by default. Keep those as explicit upgrade paths triggered by scale, reliability, multi-host, direct browser transfer, or compliance needs.

The baseline is viable only if it is treated as an operational contract, not a dumping folder. Every artifact write needs an app-generated key, checksum, size, MIME evidence, retention class, sensitivity state, and lifecycle status. Every backup needs both Postgres metadata and the artifact bytes from the same backup set.

Use a storage adapter boundary immediately: local_fs now and s3 later. The database row format should not change when the backend changes.

Scope And Method

Four sub-agents researched independent slices: artifact model and serving, Docker Compose topology, backup/retention/cleanup, and migration/upgrade triggers. Sources favored official Docker, Temporal, PostgreSQL, FastAPI/Starlette, LiteLLM, AWS S3, MinIO, OWASP, and prior local decision artifacts.

Prior DecisionConstraint For This Baseline
FastAPI and Postgres own product state. Artifact metadata, ACLs, retention, audit, backup references, and UI state are app tables and APIs.
Temporal owns execution state. Temporal stores small claim-check references only. Prompts, traces, CLI logs, diffs, and generated files stay outside event history.
React/Vite is the product UI. Static build can be served by proxy/API in deployment; Vite dev server is local-only.
LiteLLM Proxy is the model gateway. Compose includes LiteLLM with healthcheck and secrets. Postgres-backed mode is useful if virtual keys, spend, or model DB are enabled.
Codex/Claude Code adapters are subscription-first CLI lanes. Workspaces should be host-visible bind mounts in dev; coding evidence becomes controlled artifacts, not arbitrary filesystem sharing.
03

Artifact Architecture

The activity receives or produces bytes, the artifact service reserves a row and writes immutable bytes, then returns a compact claim-check reference. FastAPI serves private artifacts through authenticated routes; static mounts are only for public app assets.

CategoryExamplesDefault Retention
run_input Task packet, prompt/context snapshot, spec/registry snapshot hashes. 180-365 days, longer for accepted deliverables.
agent_trace Model messages, event JSONL, tool-call stream, CLI stdout/stderr. 30 days, or 7 days if unredacted/sensitive.
tool_result Structured outputs, external API responses, generated data files. 30-180 days depending on sensitivity and replay value.
coding_evidence Git diffs, untracked captures, test logs, build logs, review findings. 180-365 days for accepted work; short retention for failed attempts.
deliverable Reports, HTML artifacts, exports, patches, images, PDFs. Project lifetime unless explicitly deleted.
preview Redacted excerpts, thumbnails, safe rendered previews. Derived; delete with source artifact or regenerate.
audit_security Redaction report, checksum manifest, secret-grant usage, scan results. 1-2 years or project lifetime.

05

Backup, Cleanup, And Serving

ConcernV1 DecisionReason
Write flow Reserve row, write temp file on same filesystem, compute SHA-256/size/type, atomic rename, mark available. Filesystem and DB are not one transaction; lifecycle status and sweeper handle failures.
Serving Use authenticated FastAPI routes and Starlette/FastAPI file responses for downloads/previews. StaticFiles is for public app assets, not per-artifact ACLs or preview policy.
Preview safety Allowlisted preview types only. Unsafe HTML, SVG, markdown, and logs default to download or sandboxed/redacted preview. Previews are a security boundary, not only a UX feature.
Backup Quiesced v1 backup window: maintenance-mode API/workers, pg_dump -Fc, artifact archive, manifest/checksums, one backup_set_id. DB metadata and files must restore to a coherent point.
Restore Periodic fresh restore drill: restore DB dump, restore artifact archive, run checksum and orphan scans. Until restore is tested, backup is only a hope.
Cleanup DB-driven retention job skips active runs/holds, records tombstones, deletes bytes, and runs orphan scans both directions. Scanning filesystem should not define product truth.
Secrets Never store tokens, DB URLs, OAuth material, credential files, or full env dumps as artifacts. Use Compose secrets and app redaction; secret values do not belong in evidence storage.

Decision Matrix

OptionBest FitSacrificeVerdict
Local filesystem + Postgres metadata Single-host v1, local CLI adapters, fast iteration, simple backups if disciplined. No multi-host artifact sharing, manual backup discipline, disk pressure risk. Choose v1
Postgres blobs or large objects Small binary fields tightly coupled to relational rows. Bloats DB backups, locking, and restore path for large traces/logs/files. Reject for artifacts
MinIO in default Compose Early testing of S3 SDK behavior, presigned URLs, multipart, lifecycle semantics. Another stateful service before object-store pressure exists. Optional profile
S3/object storage from day one Multi-host, direct browser transfer, lifecycle policy, cloud durability. Credentials, IAM/policy, CORS, endpoint, cost, and local-dev friction. Upgrade path
Docker Compose deployment Trusted single host, local dev, prototype, early internal use. No HA, weak rolling deploy story, host-level backup and secrets responsibility. Choose baseline
Kubernetes/managed services now Multi-host production, autoscaling, managed secrets, separate worker pools, stronger ingress. Large operations jump before product pressure justifies it. Defer
07

Upgrade Triggers

Move ToTriggerPreparation Already Required In V1
S3/object storage Workers on multiple hosts, direct browser upload/download, local backup discomfort, disk pressure, customer durability, lifecycle/retention needs. Backend field, S3-safe keys, SHA-256/size metadata, no local-path authority.
MinIO local profile You need to test S3 semantics before cloud: presigned URLs, multipart, bucket lifecycle, SDK compatibility. Storage adapter tests that run against both local filesystem and S3-compatible backend.
Temporal Cloud Workflow availability matters beyond the Compose host, Temporal upgrades/security/visibility become a tax, or multiple worker fleets need managed connectivity. Keep Temporal endpoint/config injectable; do not couple code to local service names.
Managed Postgres Required PITR, failover, automated backups, patching, storage scaling, monitored performance, or meaningful user/customer data. Use standard PostgreSQL backup/restore and migrations, not container-volume assumptions.
Container orchestration More than one host, rolling deploys, autoscaling, separate worker pools, network policy, managed secrets, production ingress. Clear service boundaries, healthchecks, env/secrets discipline, stateless API/worker containers.
Artifact service Preview transforms, scanning, quotas, signed-link policy, CDN, cross-runtime sharing, tenant-heavy audit, or dedicated artifact permissions. Keep FastAPI artifact routes behind an internal service boundary and stable artifact IDs.
08

Failure Modes

RiskSeverityMitigation
Postgres metadata and artifact bytes restore to different points. High Quiesced backup window, shared backup_set_id, manifest/checksums, restore drill.
Live Postgres volume tar produces an unusable or inconsistent backup. High Use pg_dump for small v1; move to physical backup/WAL/PITR when RPO/RTO requires it.
Secrets, OAuth tokens, DB URLs, or provider keys leak into logs/artifacts. High Denylist, redaction, Compose secrets, env discipline, never-artifact secret classes.
Preview route creates XSS or serves unsafe active content. High Allowlisted preview types, sandbox/redaction, default-to-download for unknown HTML/SVG/markdown/logs.
Temporal event history bloats with large prompts, traces, or files. High Claim-check references only; large payloads go to artifacts with hashes and summaries.
Host bind mount lets a compromised container damage host files. High Dedicated host dirs, non-root UID/GID, read-only mounts where possible, least mounted services.
Disk fills from traces, screenshots, or video/debug bundles. Medium Category retention, file caps, quotas/alerts, cleanup job, large-debug retention class.
Compose debug ports expose Temporal, LiteLLM, Postgres, or UI publicly. Medium Publish only proxy by default; local debug ports bind 127.0.0.1.
