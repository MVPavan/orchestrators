Final minimal architecture

Consolidated orchestrator report from sub-agent research across the resolved lean AI-company architecture decisions: FastAPI, React/Vite, Temporal, Pydantic AI, LiteLLM, app-owned registries, subscription-compatible coding adapters, local artifacts, and Docker Compose.

FastAPI + React FastAPI owns API/control contracts. React/Vite owns the cockpit and consumes app-owned APIs and event streams.
Temporal + Postgres Temporal owns durable execution. Postgres owns product truth, audit, specs, tools, approvals, artifacts, and UI-visible state.
Pydantic AI + LiteLLM Pydantic AI is the default Python SDK adapter. LiteLLM Proxy is the model gateway, not the agent state layer.
CLI coding lanes Codex CLI and Claude Code CLI are the subscription-first coding adapters. SDK paths stay conditional.
01

Answer

Decision: build the first version as a FastAPI control plane, React/Vite cockpit, app-owned Postgres product database, Temporal execution engine, Pydantic AI agent adapter, LiteLLM model gateway, app-owned Capability Registry and Spec Registry, subscription-compatible Codex and Claude Code CLI coding adapters, local claim-check artifact storage, and Docker Compose single-host deployment.

V1 discipline: prove one vertical slice before broadening the platform: one run, one role snapshot, one model alias, one gated tool, one approval, one artifact, one event stream, and one contract-test harness.

The important boundary is not which library has the broadest feature list. It is which component owns which truth. Postgres is product and audit truth. Temporal is execution truth. LiteLLM is model gateway truth. Pydantic AI, Codex, Claude Code, MCP, and OpenAI/LangChain integrations are adapter surfaces. React is a projection and control surface, not durable workflow state.

The corrected coding-adapter stance is CLI-first for subscription compatibility: Codex uses codex exec, codex exec resume, and codex exec review; Claude uses claude -p for the subscription-OAuth path. Claude Agent SDK is conditional, not the default, unless its auth and billing model is explicitly acceptable for the deployment.

05

Integration Path

  1. A user starts a run through FastAPI. FastAPI authenticates the user, resolves active spec/tool/model/approval/adapter versions, writes an app run and immutable run snapshot, then starts a Temporal workflow with compact IDs and hashes.
  2. Temporal executes deterministic workflow logic. Any model call, DB write, file operation, tool call, CLI process, or external network call happens through activities or supported durable SDK integration.
  3. Pydantic AI is constructed from the per-run snapshot and calls LiteLLM Proxy for model access. LiteLLM handles provider aliases and model gateway behavior; it does not become session memory or artifact storage.
  4. Tool calls flow through generated per-run toolsets into the Tool Execution Gateway. The gateway validates exact tool version, input schema, args hash, actor, role, scope, and required approval.
  5. Human approvals are app-owned rows in Postgres. Temporal waits on a signal or update tied to the exact approval row, tool version, args hash, actor, scope, and expiry.
  6. Large outputs use claim-check storage. Activities reserve artifact rows, write immutable local bytes, calculate checksum/size/MIME, and return compact artifact references.
  7. FastAPI projects run status, events, approvals, and artifacts to React through REST and SSE. The browser may cache and filter state, but app APIs remain authoritative.
  8. Coding work is a later specialized workflow. A coding task leases an isolated git worktree, invokes Codex or Claude Code through subscription-compatible CLI adapters, captures logs/diffs/tests/artifacts, and requires app policy or human approval before apply.

Upgrade Triggers

Do Not Add By DefaultAdd WhenKeep The Exit Path Ready By
Next.js Public SEO pages, SSR, edge rendering, or a Node BFF becomes a real requirement. Keeping the UI behind stable FastAPI APIs and generated contracts.
HTMX as default The UI stays under a few simple internal screens and rich timelines/diffs/streaming controls do not materialize. Keeping React screens as API consumers rather than business logic owners.
Prefect Scheduled data/eval/ops jobs need Prefect's operator experience outside the core human-gated run lifecycle. Keeping Temporal core workflows and app-owned run state separate from batch job orchestration.
OpenAI Agents SDK as central SDK Its provider behavior, LiteLLM path, tracing, tools, and handoffs pass the app's contract gates better than Pydantic AI for a specific lane. Treating it as a sidecar adapter behind app-owned specs/tools/state.
LangChain DeepAgents or LangGraph as central runtime A specific multi-step research/agent pattern needs its graph abstractions and passes artifact/tool/approval boundaries. Invoking it inside a Temporal activity or sidecar lane, not replacing the app-owned workflow boundary.
Claude Agent SDK API-key billing is acceptable, or Anthropic-supported subscription-product auth is explicitly available for the deployment. Keeping Claude Code CLI claude -p as the subscription-first v1 path.
MCP server marketplace The app-owned Tool Execution Gateway exists and external runtimes need a portable tools/list and tools/call ABI. Maintaining canonical tool definitions, policy, approvals, secrets, and audit in Postgres.
S3 or MinIO Multi-host workers, disk pressure, direct browser uploads/downloads, object lifecycle policy, or backup requirements exceed local files. Using object-shaped keys, storage_backend, checksums, and claim-check refs from day one.
Temporal Cloud or managed Postgres Availability, failover, PITR, upgrades, security maintenance, or operator burden exceed the single-host baseline. Keeping endpoints/config injectable and restore drills explicit.
Kubernetes Multiple hosts, independent worker pools, rolling deploys, resource isolation, or production SRE controls are necessary. Avoiding Compose-specific assumptions in app code.
08

Risks And Gates

RiskSeverityGate Or Mitigation
Temporal and Postgres diverge on run state after duplicate starts, failed signals, cancellation, or worker replay. High Define an app run state machine, Temporal state mapping, idempotency keys, outbox/projector, reconciliation job, and operator repair path.
Subscription-compatible coding adapters are treated as if one server-side desktop login can safely serve all users. High Model adapter modes explicitly: single-user local runner, per-user local runner, service credential runner, and API-key fallback. Gate each by policy.
FastAPI auth/admin is deferred while later layers assume enforcement exists. High Pick a v1 auth posture before real workflows: external OIDC/Authlib plus server-side session/CSRF, or explicitly single-operator local-only.
Tool registry becomes a marketplace before one workflow proves schemas, approvals, secrets, and audit. Medium Seed only a small curated registry. No raw MCP/OpenAPI import until the app-owned executor and policy checks work.
LiteLLM fallback or provider changes break tool calling, strict output, streaming, or usage reporting. High Activation depends on model/tool contract tests per alias and SDK path.
Backups capture app DB and artifacts but not Temporal or LiteLLM persistence, creating restore ambiguity. Medium Define backup sets that include app DB, Temporal DB, LiteLLM DB if enabled, and artifact bytes. Run restore drills with workers/API quiesced.
Coding runs trust final agent prose instead of supervisor-verifiable diffs, test output, and artifacts. Medium Supervisor captures git diff, command logs, test results, artifact hashes, and critic output before human apply/discard.
