UI layer for a FastAPI assistant product

Consolidated sub-agent research on the UI approach for a FastAPI-backed personal assistant and multi-agent control plane, after the prior decision to use FastAPI as the core API layer.

React + Vite Recommended default for a real assistant cockpit over FastAPI APIs.
HTMX fallback Best for a small internal MVP with forms, lists, and simple streams.
Skip Next.js Until SSR, public pages, or a deliberate Node BFF is a real requirement.
Prototype tools Streamlit, Gradio, NiceGUI, Panel, and Dash are bounded surfaces, not the product shell.
01

Answer

Recommendation: build the main product UI as a React + Vite + TypeScript SPA over FastAPI APIs, preferably same-origin behind the same reverse proxy or served as built static assets by the FastAPI deployment.

Reason: the named workflows are not just CRUD: assistant chat, workflow timelines, approval queues, artifact review, code diffs, streaming run status, and future multi-agent observability need reusable client state, dense components, rich diff/artifact rendering, and a mature browser UI ecosystem.

The previous research decision fixed FastAPI as the backend/API edge. Django is therefore not a live framework candidate here except as a reminder that server-rendered/admin-heavy products can be simpler in a Django-style stack. For this FastAPI path, Next.js should not be introduced unless the product explicitly needs SSR/public web, React Server Components, or a Node BFF.

Architecture Guidance

Keep FastAPI authoritative

  • FastAPI owns auth/session enforcement, OpenAPI, workflow APIs, artifact metadata, approval actions, and stream endpoints.
  • Postgres/workers own durable state; the browser displays and edits through APIs.
  • Generate TypeScript client/types from `/openapi.json` and fail CI if generated artifacts drift once CI exists.
  • Prefer same-origin deployment early to avoid avoidable CORS and cookie complexity.

Transport defaults

  • SSE/EventSource for one-way run status, token/status streams, queue updates, and artifact progress.
  • WebSocket only for frequent bidirectional control, live terminal/editor behavior, collaborative controls, or run steering.
  • Design reconnect, `Last-Event-ID`/resume, cancellation cleanup, and multi-worker fanout before scaling beyond one web process.
  • Do not make frontend state the source of truth for workflow progress.

Counterarguments And Risks

The strongest case against React + Vite is real: it adds Node tooling, frontend skill demands, generated client discipline, API state management, accessibility work, and frontend tests. If the first release is a single-operator cockpit with fewer than five simple screens, no real diff viewer, and no dense artifact review, HTMX will ship faster and stay closer to FastAPI.

RiskSeverityMitigation
React overbuilds an internal-only MVP. Medium Start with a thin route set, TanStack Query only for server state, and postpone complex global state.
Frontend duplicates workflow state and creates split truth. High Keep Postgres/workers authoritative; browser state is selection, filters, optimistic UI, and stream buffers only.
Next.js creeps into backend responsibilities. High Do not add Next.js unless SSR/public web or a Node BFF is explicitly approved.
SSE/WebSocket reconnect and fanout bugs lose status events. Medium Use server-side event history, resume tokens, and broker/event-log fanout before multi-worker scale.
Prototype tools become accidental product shell. Medium Mount Streamlit/Gradio/NiceGUI only as bounded internal sub-apps with clear retirement criteria.
Diff libraries solve rendering but not review semantics. Medium Plan file tree, comments, permissions, patch application, large-diff handling, and audit trails as product features.

TriggerMoveReason
