Tool registry and SDK tool integration strategy

Consolidated sub-agent research on building an internal tool registry for the FastAPI, Postgres, Temporal, Pydantic AI, LiteLLM, Codex CLI, and Claude Code CLI architecture.

Registry is product state Store canonical tool definitions, permissions, approvals, audit, and versions in Postgres.
Pydantic AI is adapter Generate stable toolsets from registry snapshots; do not let SDK objects become truth.
MCP is ABI Use MCP as the portability interface for external ecosystems, not as the policy source.
Approvals are durable Approval rows and argument hashes live in the app, not transient SDK callback state.
01

Answer

Recommendation: build a small app-owned Capability Registry plus Tool Execution Gateway in FastAPI/Postgres. Expose per-run, policy-filtered toolsets to Pydantic AI. Expose the same curated subset as internal MCP only when another runtime needs portability.

Do not make Pydantic AI toolsets, MCP server listings, OpenAPI imports, LangChain tools, LiteLLM aliases, Claude permissions, or OpenAI Agents SDK tool catalogs the registry of record. Those are adapter surfaces.

The internal registry should feel like a small internet-style tool catalog: searchable names, namespaces, versions, descriptions, examples, owners, risk badges, required grants, and role availability. But execution must be conservative: no raw external catalog is automatically callable. A tool becomes callable only after curation, schema validation, policy assignment, approval policy, secret binding, and contract tests.

For v1, start with native Python registry tools exposed through Pydantic AI FunctionToolset or a custom registry-backed toolset, plus a few hand-wrapped MCP/CLI adapters. Defer a broad plugin marketplace, LiteLLM MCP Gateway as a security boundary, and LangChain/DeepAgents as the central registry until concrete gaps appear.

Subscription-login gate: every adapter must be checked for whether it can run under the required account model, not just whether it accepts API keys. Claude Code CLI is the clear subscription-compatible coding adapter. Pydantic AI, OpenAI Agents SDK, LangChain/DeepAgents, and LiteLLM are primarily SDK/API-key or gateway surfaces for service-owned model traffic. MCP is auth-neutral transport: each MCP client/server pair still needs its own subscription-login or API-key validation. Claude Agent SDK remains conditional until its current auth and billing boundary is verified for the intended deployment.

Context And Method

Research was split across four sub-agents: Pydantic AI tool mechanics, ecosystem reuse across OpenAI/Claude/LangChain/DeepAgents/LiteLLM, registry and security architecture, and red-team synthesis. Source weighting favored current official docs, current specs, and prior local decision artifacts.

Prior DecisionConstraint For Tool Strategy
FastAPI and Postgres own product state. Registry, approvals, tool calls, artifacts, audit, and UI event projection are app tables and APIs.
Pydantic AI is the default in-process SDK. Use Pydantic AI as the primary tool adapter, but keep canonical definitions outside SDK objects.
Temporal owns durable workflow execution. Tool execution runs in activities or integration-managed activities; workflows use stable registry snapshots and claim-check payloads.
LiteLLM Proxy is the model gateway. Run contract tests per model alias for tool calling, schema strictness, streaming, usage, and fallback behavior.
Codex and Claude are subscription-first CLI adapters. Expose them as controlled coding-run tools or child workflows, not as generic shell access or SDK-owned tool registries.

04

Registry Contract

RecordRequired FieldsWhy It Exists
tool_definition tool_id, namespace, name, version, owner, status, source, transport, schema dialect, input schema, output schema, content hash. Canonical tool identity and contract. Versions and hashes prevent silent schema mutation.
tool_policy risk class, side-effect class, role allowlist, tenant/resource scope, approval policy, sandbox policy, network policy. Separates capability description from authorization and safety.
tool_runtime executor kind, endpoint/ref, timeout, retry policy, idempotency key template, concurrency/sequential flag, streaming support. Lets the same catalog support Python functions, MCP, HTTP APIs, CLI adapters, and child workflows.
tool_secret_binding secret aliases, grant scope, lease duration, injection mode, redaction policy. Secrets are never model-visible inputs. Executors resolve short-lived grants server-side.
tool_call run id, step id, tool version, validated args hash, approval id, executor id, status, retry count, started/ended timestamps. Durable audit and replay boundary for every invocation.
tool_result ok, result version, data or artifact refs, redaction status, external ids, retryable, safe-to-show flag, audit severity. Gives the model a sanitized result while the app retains full provenance.

Error Taxonomy

Normalize errors into protocol/contract, validation, auth/grant, approval denied, rate limit, sandbox violation, timeout/cancelled, upstream failure, retry exhausted, output validation, and internal defect. Each error should state whether it is retryable, user-correctable, safe to show the model, and whether it requires audit escalation.

Execution Lifecycle

StepOwnerRule
Resolve FastAPI + Postgres Resolve exact tool version and registry snapshot for the run. Do not bind to "latest" inside a durable workflow.
Authorize FastAPI policy layer Filter by role, department, tenant, resource scope, and model/runtime capability before presenting tools.
Validate Gateway Validate input against registry JSON Schema/Pydantic model; reject unknown fields for sensitive tools.
Approve Postgres + UI + Temporal signal/update Approval is keyed to exact tool id, version, args hash, actor, scope, and expiry. Changed args invalidate approval.
Execute Temporal activity or controlled adapter External I/O, model calls, MCP calls, DB queries, file I/O, and CLI runs stay out of deterministic workflow code.
Persist Postgres + artifact store Store normalized result envelope and large payloads by reference. Avoid large prompt/tool payloads in Temporal history.
Project FastAPI event stream Project stable app events to the UI. SDK streams are input signals, not the durable user-facing ledger.

Failure Modes

RiskSeverityMitigation
SDK object graph becomes the product registry. High Persist canonical registry records in Postgres; generate SDK toolsets from immutable snapshots.
External MCP/OpenAPI catalog is exposed wholesale. High Require curation, policy assignment, contract tests, and per-run filtering before exposure.
Approval is tied to a tool name, not exact arguments. High Approval rows include tool version, args hash, scope, reviewer, expiry, and decision. Changed args re-prompt.
Secrets leak through model context, SDK traces, MCP metadata, or Temporal history. High Use aliases and short-lived server-side grants only. Redact before persistence and never put raw secrets in tool args.
Temporal workflows replay non-deterministic tool or network calls. High Run model/tool/MCP/API/file I/O in activities or supported integrations; keep workflow code deterministic.
Tool schema mutates while a long-running workflow waits for approval. Medium Pin registry snapshots and route old workflow versions until they drain or migrate explicitly.
LiteLLM/provider fallback changes tool or structured-output behavior. Medium Run provider contract tests for tool calls, strict schemas, streams, usage, and fallback behavior before enabling aliases.
Registry becomes a bloated marketplace before first workflows exist. Medium Start with 8-12 curated internal tools and add external imports only when a workflow needs them.
08

Implementation Decisions

QuestionDecisionFirst Spike
What is the first registry type? Postgres tables plus FastAPI registry/execution endpoints. No separate registry service in v1. Create ToolDefinition, ToolPolicy, ToolCall, ToolResult, and Approval tables.
How does Pydantic AI see tools? Generate per-run Pydantic toolsets from registry snapshots and role policy. Build one registry-backed FunctionToolset wrapper with one read tool, one approval tool, and one artifact-returning tool.
How do other runtimes reuse tools? Expose a curated internal MCP server after the app-owned executor exists. Serve tools/list and tools/call for the same approved subset; consume it from Claude CLI or OpenAI Agents SDK.
How are human approvals represented? App approval row is authoritative; SDK approval/deferred-tool APIs are adapters into that row. Pause one tool call, write approval row, approve via FastAPI, and resume Temporal workflow.
What tools are custom first? Run status, artifact search/read, approval request, memory/RAG read, coding-run launch, web fetch/search, and external API wrappers. Do not expose generic shell, browser, filesystem write, or broad MCP servers until sandbox/secret policy exists.
