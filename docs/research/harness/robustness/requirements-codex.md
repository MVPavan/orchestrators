# Requirements for Understanding a Codebase — Codex (gpt-5.5) first-principles catalogue

_Tool-agnostic, independent of the Opus pass and of scenarios.md. Model: gpt-5.5, effort xhigh._

**Framing**

A person or agent “understands” a codebase when they can repeatedly answer four questions with evidence and bounded effort:

1. Where is the relevant thing?
2. What does it do, and why?
3. What will happen if I change it?
4. How do I prove the change is safe?

A good codebase-understanding method is not “more documentation” or “a prettier graph.” It is a capability that reduces time, error rate, and uncertainty for real changes.

Below, each objective test uses this shape: `Q = question to pose; Gold = ground truth; Metric = pass criterion`.

| 15 | **Business Rules**: Identify policy decisions, thresholds, eligibility rules, and exceptions. | These are often the real behavior, scattered across code and config. | Humans ask domain experts; agents need cited rule extraction. | Q: “When is action A allowed?” Gold: tests/specs/code paths. | 16 | **Design Rationale**: Explain why code is shaped this way, including rejected alternatives. | Without rationale, “cleanup” can reintroduce old failures. | Humans ask teammates; agents need links to commits, issues, docs, comments. | Q: “Why not simplify module M?” Gold: design record or maintainer judgment. | 17 | **Naming/Alias Resolution**: Connect synonyms, renamed concepts, abbreviations, legacy names. | Large systems accumulate multiple names for one idea. | Humans infer aliases; agents need alias tables and uncertainty labels. | Q: “Are A, B, and C the same concept?” Gold: schema/history/maintainer. | 18 | **Legacy/Deprecated Path Detection**: Identify dead, old, experimental, migration-only, or compatibility code. | Prevents fixing unused code or copying obsolete patterns. | Humans use tribal knowledge; agents need evidence from references, flags, runtime, history. | Q: “Is path P live?” Gold: production config, traces, references, owners. | 19 | **User Behavior Mapping**: Connect screens/endpoints/commands to user outcomes. | Code changes should be evaluated against behavior, not files. | Humans can run product flows; agents need scenario-to-artifact mapping. | Q: “Which user journeys use component C?” Gold: routing/tests/analytics/specs. | 20 | **Data Model**: Understand entities, schemas, types, constraints, indexes, migrations. | Most bugs are data-shape bugs. | Humans can inspect ER diagrams; agents need exact schemas and type contracts. | Q: “What fields define entity E?” Gold: schema/migration/type source. | 21 | **Data Lineage**: Trace where data originates, transforms, validates, persists, and is read. | Prevents fixing symptoms downstream while corrupting upstream assumptions. | Humans draw flow diagrams; agents need compact lineage chains. | Q: “Where does field F come from and who consumes it?” Gold: source paths plus runtime traces. | 22 | **State Lifecycle**: Know what state is durable, cached, derived, ephemeral, synchronized, or eventually consistent. | Wrong state assumptions cause race conditions and stale reads. | Humans reason with diagrams; agents need state classification. | Q: “After event E, when is state S updated?” Gold: code + tests + traces. | 25 | **Trust/Security Boundary**: Locate authentication, authorization, validation, encryption, PII, sandboxing, privilege changes. | Security bugs come from misunderstanding boundaries. | Humans may rely on review norms; agents need explicit high-risk markers. | Q: “Where is permission P enforced?” Gold: security tests/code review answer. | 29 | **Runtime Lifecycle**: Know startup, initialization, request handling, shutdown, workers, migrations. | Static structure misses temporal behavior. | Humans observe logs; agents need ordered lifecycle with entrypoints. | Q: “What starts first and what owns shutdown?” Gold: runtime trace/source. | 43 | **Change Hotspots**: Identify frequently modified, defect-prone, or ownership-heavy areas. | Guides caution and review depth. | Humans remember painful files; agents need history-derived risk. | Q: “Which files are riskiest for task T?” Gold: commit/bug history and maintainer labels. | 47 | **Cost/Latency Efficiency**: Answer with minimal reading, tokens, commands, and wall time. | Understanding must be economical under daily use. | Humans pay attention cost; agents pay tokens/context and command latency. | Q: “Answer 50 navigation questions.” Gold: answer set. **Prioritized View**

Most load-bearing for day-to-day productivity:

1. **Feature-to-code mapping**, **exact symbol/reference traversal**, and **intent search**: without these, everything starts too slowly.
2. **Runtime/control-flow tracing**, **data lineage**, and **configuration model**: these prevent structurally plausible but behaviorally wrong edits.
3. **Build/run/setup**, **test map**, and **verification selection**: these turn understanding into safe change.
4. **Impact/blast-radius analysis**, **contracts**, and **invariants**: these are the core safety checks before editing shared code.
5. **Provenance and uncertainty handling**: these determine whether a human should trust the answer.

Most often under-served:

1. **Intent and rationale**: typical approaches find code, but not why it exists.
2. **Runtime behavior**: static maps miss initialization order, async work, retries, and config-dependent behavior.
3. **Data lineage and state lifecycle**: code search often finds functions but not how data mutates across time.
4. **Operational reality**: deploy, rollback, migrations, alerts, and incidents are usually outside code navigation systems.
5. **Verification selection**: many systems can list tests; fewer can choose the right proof for a specific change.
6. **Uncertainty labeling**: most tools return answers without saying what is inferred, stale, or unsupported.
7. **Cost efficiency for agents**: many methods are acceptable for human browsing but too verbose, imprecise, or expensive for token-limited agents.

**Easy-To-Overlook Requirements**

The most dangerous omissions are not directory trees or call graphs. They are comprehension layers: domain rules, design rationale, invariants, runtime lifecycle, configuration precedence, external failure behavior, operational rollback, and evidence quality. A method that cannot answer those may help someone move around the repository, but it does not yet help them change the system safely.
