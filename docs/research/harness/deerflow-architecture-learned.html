Learning DeerFlow 2.0

A working understanding of the codebase, produced by actually running two AI code-understanding tools in their intended modes — graphify (offline symbol graph + docstring extraction + graph queries) and understand‑anything (multi-agent file-analysis → layered architecture → guided tour). Not a parse benchmark — this is what the tools teach.

TL;DR — what DeerFlow is. DeerFlow 2.0 is a multi-channel, agentic assistant platform. Users talk to it from ordinary IM apps — WeChat, DingTalk, Feishu, Telegram, Slack, Discord, WeCom — and each message flows through an in-process MessageBus (async pub/sub) into a LangGraph-based agent harness that looks a lot like an open-source Claude‑Code: a deep middleware stack, recursive sub-agents, installable skills (slash-commands), a code-execution sandbox, durable memory, MCP tools, and a dozen web-search providers. A FastAPI gateway exposes the harness over HTTP/SSE (LangGraph-Platform-compatible), and a Next.js/React frontend renders the streaming, "show-your-work" agent UI. The real engineering mass lives in the harness; channels and UI are comparatively thin edges.

3 · Guided onboarding tour — read DeerFlow in this order

understand-anything's tour-builder walked the real dependency graph (BFS from the gateway entry point, foundation-first) to produce a 14-step reading order. Start at the top:

  1. Gateway entry point — backend/app/gateway/app.py
    The FastAPI factory that mounts every router and boots the LangGraph runtime singletons. The system's front door.
  2. Auth & request identity — gateway/auth_middleware.py · authz.py
    Fail-closed JWT auth + permission/ownership decorators that gate every route (cross-user access returns anti-enumeration 404s).
  3. Dependency-injection hub — gateway/deps.py
    How a stateless HTTP request is wired to the stateful runtime singletons on app.state.
  4. Configuration foundation — config/app_config.py
    The most depended-upon module: AppConfig aggregates ~30 domain sections from one config.yaml, with content-signature hot-reload.
  5. Per-user scoping — runtime/user_context.py
    The ContextVar identity that scopes every sandbox, storage path, and DB query per user.
  6. In-process client & lead agent — client.py · agents/lead_agent/agent.py
    The agent core: a config-driven factory that stacks the middleware chain on langchain's create_agent. DeerFlowClient is the no-HTTP, gateway-equivalent surface.
  7. Agent state model — agents/thread_state.py
    The typed ThreadState (sandbox/title/artifacts/todos/uploads/…) with LangGraph reducers — the channel through which middlewares communicate.
  8. Skills: activation & storage — skills/types.py · skills/storage · skill_activation_middleware.py
    /skill-name activation injects a SKILL.md as hidden context; skills also gate which tools an agent may call.
  9. Sub-agents & delegation — tools/builtins/task_tool.py · subagents/executor.py
    Recursive multi-agent delegation runs on isolated event-loop threads; the backend polls every 5s and streams task_started/running/completed. Sub-agents can't nest.
  10. Sandbox & MCP tools — sandbox/sandbox.py · sandbox/tools.py · mcp/tools.py
    The capability surface: per-thread virtual-path translation (/mnt/user-data/…), traversal/bash-allowlist defense, and session-pooled MCP tools.
  11. Memory subsystem — agents/memory/updater.py · storage.py
    LLM-driven, debounced write-back of durable facts; atomic per-user JSON; token-budgeted injection that reserves room for user corrections.
  12. Run execution & streaming — runtime/runs/worker.py · gateway/services.py · routers/thread_runs.py
    Where an HTTP request becomes a live SSE-streaming agent run via the resumable StreamBridge.
  13. Persistence & thread metadata — persistence/base.py · thread_meta · run/sql.py
    The SQLAlchemy backbone (runs/threads/events) that re-enforces ownership at the data layer.
  14. Frontend client — frontend/src/core/api · workspace/chats/chat-box.tsx
    The Next.js end of the same request path that began at app.py — closing the loop.

4 · Subsystem deep-dives

Grounded summaries from understand-anything's per-file analysis (12 core batches, 304 files) and graphify's docstring "rationale" extraction.

Layer 2 · the heart

Agent core & middleware stack

Two-layer factory: create_deerflow_agent (config-free SDK entry, flags → ordered middleware via @Next/@Prev anchors) and make_lead_agent (config-driven app factory) stack a ~19-stage chain on langchain's create_agent. Almost all agent behavior is cross-cutting LangGraph hooks, not agent logic:

loop-detection (P0 hard-stop)input-sanitization (prompt-injection)sandbox-audit (bash)safety-termination detectorssummarizationtoken-budget / token-usagesubagent-limit [2,4]skill-activationtodo / plan-modedynamic-context (date+memory)auto-titleLLM-error retry/backoff

Layer 2/3 · delegation

Sub-agents & tools

get_available_tools centralizes assembly: config tools (reflection) + builtins (present_files, ask_clarification, view_image, setup_agent) + MCP + ACP agents + the task sub-agent tool. task_tool → SubagentExecutor.execute_async builds a fresh LangGraph agent on a dedicated daemon-thread event loop (run_coroutine_threadsafe), so delegation never blocks the parent. Sub-agents run with checkpointer=False and cannot nest.

Layer 3 · safety surface

Sandbox

One abstract Sandbox ABC (execute_command / read·write·glob·grep), two interchangeable backends chosen by config: host-fs LocalSandboxProvider and Docker AioSandboxProvider. The agent always sees stable virtual paths (/mnt/user-data/{workspace,uploads,outputs}, /mnt/skills) mapped to per-(user,thread) physical dirs. Defense-in-depth in sandbox/tools.py: traversal validation, bash allowlist, host-path masking in output. The AIO provider is a stateful container pool with cross-process file locks, health-gated adoption, and idle reaping.

Layer 2 · durable state

Memory

An async, debounced, LLM-driven write-back loop: MemoryMiddleware.after_agent filters the conversation, detects EN/ZH correction/reinforcement signals, and enqueues on a process-global MemoryUpdateQueue (30s debounce, dedup per thread/user/agent). A timer drives MemoryUpdater, which prompts an LLM to extract facts & context updates. Storage = atomic per-user/agent JSON (FileMemoryStorage, swappable). On read, facts are packed under a token budget with a guaranteed reserve for "correction" facts so explicit user corrections are never evicted.

Layer 4 · execution

Runtime & streaming

RunManager is a lock-protected run registry over a pluggable RunStore; run_agent drives graph.astream(stream_mode=[…]). The StreamBridge ABC decouples producer from SSE consumers — MemoryStreamBridge buffers per-run events behind a condition, supports resumable subscription via last_event_id, heartbeats on idle, and signals completion. Checkpointer/store are config-selected (InMemory/SQLite/Postgres); on cancel/error the worker rolls back to the pre-run checkpoint.

Layer 5 · data

Persistence & events

Runs persist as RunRow (status, multitask strategy, denormalized token accounting) via SQL RunRepository or memory. Runtime events are a single unified journal — RunEventRow with category ∈ {message,trace,lifecycle} and a strictly-increasing per-thread seq (unique constraint) enabling cursor pagination; backends are Db / JSONL / Memory. Channel connections store OAuth state + Fernet-encrypted credentials, mapping external IM conversations to thread_ids.

Layer 3 · extensibility

Skills & MCP

Skills install with defense-in-depth: safe zip extraction (reject symlinks/traversal, zip-bomb caps) then an LLM security gate — scan_skill_content asks a chat model for an allow/block decision, defaulting to block on parse failure. tool_policy.py makes skills a per-agent tool-permission boundary (union of declared allowed-tools). MCP tools are session-pooled (LRU MCPSessionPool), sandbox-aware (rewrite file:// → virtual paths), OAuth-injected, and mtime-cached. Both configured by one extensions.json.

Layer 3 · research

Web & search tools

A uniform duck-typed contract — every provider exposes web_search_tool / web_fetch_tool / image_search_tool returning JSON. ~12 providers: bravetavilyexafirecrawlsearxngserperjina_aibrowserlessddggroundrouteinfoquestimage_search. Fetchers converge on a shared ReadabilityExtractor → Markdown. Serper adds real SSRF hardening (rejects private/loopback, normalizes octal/hex/int-encoded IPs).

Layer 1 · API

Gateway & auth

create_app() mounts ~17 routers under /api/* (models, mcp, memory, skills, artifacts, uploads, threads, agents, channels, auth, runs, …) behind Auth → CSRF → (opt-in) CORS. The ASGI lifespan builds the restart-required LangGraph singletons. Auth is two-layer fail-closed: session JWT cookie or an internal-auth header (used by IM-channel workers), else 401. thread_runs.py implements the full LangGraph-Platform-compatible lifecycle (create/stream/wait/regenerate/cancel/join) delegating to services.start_run().

Layer 1 · ingress

Channels (the front doors)

A Channel ABC with concrete adapters for WeChat, WeCom, DingTalk, Feishu, Telegram, Slack, Discord, coordinated by ChannelManager / ChannelService. Inbound platform webhooks become InboundMessages on the MessageBus; the agent's OutboundMessages stream back as platform-native cards. message_bus.py is one of the repo's largest hubs — the async seam that decouples every channel from the agent dispatcher.

Layer 6 · foundation

Configuration

AppConfig composes ~30 per-domain Pydantic sections from one config.yaml (resolution chain: explicit → env → project root → legacy), resolves $VAR env values, and hot-reloads on a sha256 content-signature change. A reload_boundary registers startup-only infra fields. Notable: tracing (LangSmith/Langfuse) is the exception — read straight from env. Many feature flags: tool_search, guardrails(.fail_closed), loop_detection, token_budget, summarization, skill_evolution, OIDC SSO.

Layer 8 · UI

Frontend

The browser talks to the backend through a memoized @langchain/langgraph-sdk client (CSRF-injected, 409-aware reconnection), not raw REST. State splits between React Context (auth, i18n) and TanStack Query (server data). The workspace is a thread-centric split-pane: message-list.tsx renders grouped assistant turns and a ToolCall renderer turns web-search / write-file calls into the agentic "show-your-work" UX and clickable artifacts; input-box.tsx (~1.3k LoC) hosts the model selector, reasoning-effort pickers, slash-command skill autocomplete, and AI follow-up suggestions. A static-mode path runs the whole app on fixtures.

6 · Stack & notable engineering

  • It is an open-source Claude-Code analog. Todo/plan-mode, sub-agents, installable skills, a sandbox, durable memory, token budgets, loop detection, prompt-injection defense, deferred tool schemas (tool_search) — the same harness vocabulary, built on LangGraph.
  • Safety is pervasive and explicit: P0 loop-detection hard-stop, provider safety-termination detectors, sandbox bash auditing + allowlist + path masking, SSRF-hardened search URLs, an LLM allow/block gate on skill install, Fernet-encrypted channel credentials, fail-closed auth.

  • Resumability & resilience: per-run checkpoint rollback on cancel, resumable SSE via last_event_id, circuit-breakered LLM retries with Retry-After honoring, crash-recovery scans for orphaned inflight runs.

7 · How the two tools learned it (and where they struggled)

graphify offline graph + queries

Method actually run

  • update → tree-sitter symbol graph (19,987 nodes, 890 communities), fully offline, no API key.
  • Mined "rationale" nodes — docstring-derived one-liners for every class/function (e.g. "MessageBus — async pub/sub hub that decouples channels from the agent dispatcher"). This alone outlined the whole architecture.
  • query / explain / hub-degree ranking — pure graph traversal, the "ask instead of grep" surface.

Limits observed in real use

  • query is keyword-BFS, not semantic — it drifted to the frontend on ambiguous terms ("reasoning", "runs").
  • export callflow-html sections are hardcoded to graphify's own repo ("Extraction Pipeline", "Graph Build", …) — useless for a foreign codebase.
  • Headless community-naming via the claude-cli backend produced 0 names in this nested-session environment; module names had to be derived from members.

understand-anything multi-agent pipeline

Method actually run

  • Deterministic scan → import-map → Louvain batching (1,327 files → 91 batches).
  • 12 file-analyzer sub-agents read 304 core files and emitted grounded summaries/tags + a 910-node, 1,955-edge graph.
  • architecture-analyzer → 8 layers; tour-builder → 14-step guided tour. These syntheses are the headline "learn the repo" value.

Limits observed in real use

  • File→file granularity: its import graph is 959 edges vs graphify's 39k symbol-level — coarser without the per-file LLM pass.
  • The frontend tsconfig.json failed to parse (JSON5 comments) → @/* path aliases unresolved, dropping some frontend edges.
  • Full LLM analysis of all 1,327 files would be costly; this run scoped to 12 of 91 batches (core backend + gateway + frontend slice).

Complementary, not redundant. graphify gave breadth instantly and for free — every symbol, every docstring, the hub map — and is the better first pass. understand-anything gave the synthesis graphify can't: layered architecture and a pedagogical reading order, at real LLM cost. Used together: graphify to map and locate, understand-anything to explain and sequence.
