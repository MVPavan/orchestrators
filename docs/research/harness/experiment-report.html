Code-Understanding Tools — what I did, in plain terms

01

What I set out to do

There are many tools that claim to help an AI assistant "understand a codebase" — by mapping it, searching it smartly, or compressing it. They don't all do the same thing, so a single head-to-head score would be unfair. The plan was:

  • Discover what each tool is genuinely built to do (read its docs and try it).
  • Use each one to its full strength against the same real codebase, and save the results.
  • Compare only where two or more tools genuinely overlap — and where all of them attempt the same thing, draw a firm conclusion.

The codebase under test, paperclip, is a real, large TypeScript project split into many packages (a "monorepo"). That's a realistic, hard target — the kind of project where naive file-by-file reading falls apart.

The one trap that made the whole thing rigorous. I picked a specific function in the code, isUuidLike (it checks whether a string looks like an ID), and asked every tool the same question: "Which files actually use this function?" I already knew the truthful answer by hand (16 files use the real shared one). That gave every tool a fair, gradeable exam — and it hid a surprise: the codebase actually contains three different functions all named isUuidLike, so a tool that just matches the name gets fooled.
02

How it was run — the team and the four phases

I acted as the orchestrator: I never did the heavy reading myself. Instead I dispatched AI workers (one per tool, working in parallel so they couldn't influence each other) to do the hands-on testing, and I used a second, independent AI critic (a different model family) to challenge the conclusions before accepting them.

Counts: 8 discovery workers + 8 usage workers + 1 final synthesis worker + 1 follow-up re-test = 18 worker runs; plus 5 critic reviews. All workers ran in the background, in parallel where possible.

03

So the tools could be compared, every "understanding" tool was asked the same set of plain questions about paperclip. Each tool answered the ones it was capable of; the rest were recorded as "out of scope by design" (which is itself a finding).

04

The 8 tools — what each is, what I found, how I used it

Each card below covers, in order: what the tool is (plain), the capabilities I discovered and then actually used, how I tested it, what it told us about paperclip, and where it fell short.

1 · ast-grep — "search by code shape"

What it is
Like find-and-replace, but it understands code structure, not just text. You write a pattern that looks like code with blanks in it (e.g. isUuidLike($$$)) and it finds every place shaped like that.
Capabilities I used
Structural search · structure-aware lint rules · a per-file outline · machine-readable output. (I deliberately did not use its code-rewrite power, since the copy was read-only.)
How I tested it
Searched for every use of the anchor; counted UI patterns across the whole app; listed a file's structure.
What it found
It cleanly separated the 24 real calls of isUuidLike from the 44 plain-text matches — it ignored the import lines and definitions that ordinary search lumps in. It also counted, in one shot: 512 UI effects, 601 UI state hooks, 412 server routes. Fastest tool of all (sub-second).
Where it fell short
It only matches shapes. It can't follow an import or tell which of the three same-named functions you mean. That's a boundary, not a bug.

2 · repomix — "pack the whole repo into one file for an AI, on a token budget"

What it is
Flattens a codebase into a single digest you can hand to an AI, and counts how many "tokens" (the units an AI reads and is billed for) each part costs.
Capabilities I used
Whole-repo pack · per-file/per-folder token counting · a compression mode · a metadata "manifest" of all files · a dependency map built from the project's package files.
How I tested it
Built a triage manifest of all 2,961 files; measured token cost; compressed one package and measured the saving.
What it found
A genuinely useful triage fact: one folder (the database package) is 77% of the entire code's token cost — and it's almost all auto-generated snapshots, i.e. "never feed that to an AI". The whole repo is ~10.9 million tokens raw — far too big for any single AI to read at once, so you must slice it. Compression saved ~58% on a code-heavy package.
Where it fell short
It prepares context; it doesn't answer questions or follow code. And its compression drops function bodies, which silently halves the accuracy of any "find usages" done on the compressed version.

3 · serena — "editor-grade go-to-definition for an AI"

What it is
Drives the very same engine your code editor uses for "go to definition" and "find all references", so an AI can navigate symbols precisely instead of guessing from text.
Capabilities I used
File symbol outline · find a symbol's definition · find references to it · go-to-definition from a usage · error diagnostics.
How I tested it
Traced the command-line entry 3 steps deep; asked "who uses the anchor?"; asked "jump from this usage to the definition".
What it found
It was the one tool that correctly noticed the anchor exists three times and refused to merge them — a precision a text search can't match. In-package navigation was clean and sub-second.
Where it fell short
"Who uses this?" returned almost nothing (1 of 16). I ran a dedicated follow-up re-test to find out why — and the cause turned out to be subtle and interesting (a "hub file" pattern), not a simple setup mistake.

4 · codegraph — "build a local map, then hand back the exact source + impact"

What it is
Scans the repo into a local database of "which function calls which", then answers a question by returning the real source lines plus what they touch — so an AI doesn't have to open 20 files.
Capabilities I used
Build the index · "explore" a topic · who-calls / who's-called · impact radius (1 hop and 3 hops) · file & index statistics.
How I tested it
Indexed 2,259 files (20 seconds); asked for the anchor's callers and its impact; explored the architecture.
What it found
It separated the three copies, followed the cross-package link with zero setup, and returned editable source plus a tiered "blast radius": changing the anchor touches 85 files at one hop, 138 at three hops. Strong at "answer me without making me read everything".
Where it fell short
It missed all 5 of the React UI usages — those live inside UI component bodies, a well-known hard case for this kind of scanner. It also indexes code only (no docs or config).

5 · codebase-memory-mcp (CBM) — "a queryable knowledge graph of the code"

What it is
Builds a graph of the whole codebase and lets you query it like a database: who-calls, call chains, impact, dead code, near-duplicates.
Capabilities I used
Index the repo · architecture overview · search · trace call paths · a read-only query language · fetch a code snippet · change-impact · near-duplicate detection.
How I tested it
Indexed 2,833 code files in 3.3 seconds; asked "who calls the shared anchor?"; ran whole-repo inventories.
What it found
best result It found all 16 cross-package callers — 100% correct — including the 5 UI usages codegraph missed, followed the workspace link with no setup, and correctly excluded the two lookalike copies. It also inventoried 1,177 routes and 15,602 functions in under a tenth of a second.
Where it fell short
Its "packages / architecture" summary is weak (it uses folder names, not real package names). Its big "10×/120× fewer tokens" marketing claims couldn't be measured from its output. And it matches by name — it would confuse two identically-named functions if their callers happened to overlap (they didn't here).

6 · graphify — "free, offline, whole-repo structure map you can query"

What it is
Builds a code graph entirely offline — no AI model, no API key, zero cost — and lets you query connections, shortest paths, and clusters.
Capabilities I used
Extract the graph · detect clusters/communities · query connections / paths / neighbours · detect import cycles.
How I tested it
Extracted 2,330 files (~40 seconds, 0 cost); asked for the anchor's connections; pulled the architecture map.
What it found
94% of the callers (15 of 16) for free, plus a weight-labelled dependency map (server→database 1,731 links, UI→shared 1,423) and real import cycles surfaced for free.
Where it fell short
real defect It silently skips any file named secrets.ts (or credential*, password*) — a built-in filter that cost it 1 of the 16 callers and is a hidden blind spot it never warns about. Offline, its clusters are left unnamed.

7 · understand-anything — "onboarding: a guided tour of an unfamiliar codebase"

What it is
Meant to teach you a new codebase with tours, a visual dashboard, and a knowledge graph.
Capabilities I used
Only the free, no-AI structural part: a file/structure scan and a call graph. The rich product (tours, dashboard, domain views) needs a paid AI model running it and a global install, which our isolation rules forbade — so it was honestly recorded as "not run".
How I tested it
Ran the free structural scan: 2,236 files in ~10 seconds, producing 9,659 functions and a call graph.
What it found
Its call graph recovered 16 of 18 callers by name (89%) for free — but its import-follower scored 0, because it ignores the workspace-link style that paperclip uses.
Where it fell short
The headline product (the interesting part) couldn't be run under isolation. We measured only its plumbing, not its promised value — and said so plainly.

8 · headroom — "shrink big tool outputs before the AI reads them"

What it is
Not a code-understanding tool at all. It's a compressor that sits in front of an AI and shrinks verbose text (logs, search dumps, JSON) to save tokens. I tested it differently and never asked it about the code.
Capabilities I used
Its compress function on four kinds of verbose text · its content-type routing · its code-aware compression.
How I tested it
Fed it four representative outputs (search results, a file listing, a source dump, a JSON array) and measured the shrinkage and whether information was lost.
What it found
JSON shrinks ~59% safely (its one trustworthy win). Search results shrink ~99% — but by throwing most matches away and keeping a sample, which would wreck a "find all usages" task. Plain source code shrinks ~0% through its normal path.
Where it fell short
On a precision task it can only lose information. It's a useful add-on for shrinking bulky structured output — not an analysis tool.

06

The re-test that corrected a conclusion (serena)

My first write-up guessed that serena scored 1-of-16 only because the test copy was stripped of its installed dependencies, and that a proper install would rescue it. Rather than leave a guess in the report, I actually ran that test: I made a fully installed copy and re-ran serena. The result was more interesting than the guess.

Forward — "where is this defined?"

fixed Broken before → now works. Once dependencies were installed, serena jumped correctly from a usage to the definition across packages.

Reverse — "who uses this?"

still 1/16 Unchanged. Installing dependencies did not help.

The real reason: paperclip exposes its shared code through a single "hub" file that re-exports everything. Serena's "find references" walks back from the definition to that hub file and stops there — it never steps out to the real users on the other side of the hub. So my original guess was only half right, and I corrected the report to say so. The honest conclusion: serena is excellent at "where is this defined?" and weak at "who uses this?" on a codebase built around hub files.

This correction was itself challenged by the independent critic and only accepted after it signed off.

07

Keeping the tools from contaminating each other

A core requirement was that installing or running one tool must not affect another, and one tool's results must never feed another's. With no containers available, I enforced this three ways:

  • Separate folders. Each tool got its own copy of the code and its own results folder. The original code was made read-only so nothing could alter it.
  • No shared installs. Every tool was installed only inside its own folder and run by its full path — never installed "globally" where another tool's worker could accidentally pick it up.
  • Private home folder. Each worker ran with its own private home directory, so anything a tool tried to write to "your account" (caches, logs, settings) landed in its own sandbox instead.

Crucially, I didn't trust the workers' own "all clean" reports — I ran an independent audit afterwards. It caught three small leaks the workers' self-reports missed, and I cleaned all three:

ToolWhat leaked into your real accountSeverityOutcome
codegraphA tiny telemetry file (a random machine ID + one "ran once" event)lowRemoved; verified gone.
graphifyOne line in a query log (a question + a file path)lowRemoved; verified gone.
serena146 setup files (during the later re-test, a background process didn't inherit the private home)lowQuarantined into the sandbox; verified gone.
None of these was a true cross-contamination — no tool became available to another, and no tool's results fed another. They were stray files in your home folder, all removed. The lesson, now recorded: the "private home" protection has to be applied from the very first time a tool is run (not just during the main phase), and it has to be passed in a way that survives background processes.

Honest limits & what's left to test

For your critical review, here is what this experiment does not prove:

Open questionStatusDetail
Did we measure the vendors' "10× fewer tokens" claims?noThose are workflow claims vs a naive read/search loop; they aren't shown in any tool's output, so they couldn't be confirmed. This is the single most valuable un-run test.
Did we run the AI-powered products of understand-anything & graphify?noTheir headline value (tours, dashboards, named clusters) needs a paid model + global install — against the isolation rules. Only the free parts were measured.
Is the comparison "tokens at equal answer quality"?noIt's accuracy-based (precision/recall on one well-understood example), not a cost-at-equal-quality benchmark across all 6 tasks.
Are codegraph's UI miss & graphify's secrets.ts bug one-offs or systemic?partlyEach is shown on one example; how often they bite across the whole repo isn't quantified.
Would serena do better on a codebase without hub files?likelyPlausible but untested — paperclip's public surface is entirely hub-exported.

One-line takeaways

  • Best all-round "understand the code" tool here: codebase-memory-mcp — most accurate cross-package answers, including the hard UI cases, at almost no cost.
  • Best free/offline option: graphify — 94% accuracy and a whole architecture map for zero tokens (mind the secrets.ts bug).
  • Best for "show me the exact code + what it touches": codegraph.
  • Best precise pattern search & safe rewrites: ast-grep.
  • Best context-prep & token triage: repomix.
  • serena — superb for "where is this defined?", weak for "who uses this?" on hub-file codebases.
  • headroom is a token-saver for bulky structured output, not an analyzer.
