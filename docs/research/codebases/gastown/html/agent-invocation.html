Agent invocation

An agent is started as a terminal process with a role-specific environment and startup prompt.

The core path is manager code to environment construction to startup command to tmux session. Work context is passed through environment variables and the startup beacon; durable assignment is read separately from Beads.

Invocation Flow

Step By Step

A command decides a session is needed.

Examples are dispatching work, booting a repo container, or a patrol loop starting a missing role session.

The role manager prepares the workspace.

Managers create or repair role directories, runtime settings, Beads redirects, hooks, and startup text before launching.

AgentEnv builds identity.

The environment sets role, repo container, worker name, session id, Beads actor, git author, and project root.

The runtime command is resolved.

Selection priority is explicit agent override, role-specific config, then default agent. Startup env is sanitized to prevent nested runtime contamination.

tmux starts the process.

Gastown creates a detached tmux session with env flags and then replaces the shell with the actual command using respawn-pane.

The agent is primed.

If hooks are available, startup hooks run gt prime. Otherwise Gastown delivers a fallback startup prompt or delayed nudge.

GT_RIGThe repo container name for repo-scoped roles.
GT_POLECAT / GT_CREWSpecific worker or crew identity when applicable.
GT_ROOTWorkspace root used by commands and routing.
GT_SESSION and GT_AGENTRuntime session and selected agent surface.
BD_ACTOR, BEADS_AGENT_NAME, GIT_AUTHOR_NAMEAttribution for Beads writes and git commits.

Actual Command Shape

The POSIX startup command is built as an exec replacement with environment entries prepended. The exact agent command varies by configured runtime.

exec env GT_ROLE=... GT_ROOT=... BD_ACTOR=... GT_AGENT=... <runtime-wrapper-if-any> <agent-command> <startup-prompt-if-supported>

The session also receives tmux -e variables before the shell starts. The duplicated env path is deliberate compatibility, not a second agent launch.

Important Safeguards

SafeguardInternal reason
Global tmux env cleanupClears stale identity variables before creating a new session so one role does not leak into another.
respawn-pane launchAvoids the race of creating a shell and then typing the startup command with send-keys.
CLAUDECODE and NODE_OPTIONS sanitizationPrevents nested agent runtimes from inheriting settings that can crash or alter behavior.
Post-create liveness checkDetects immediate command failure, kills the bad session, and returns an error instead of leaving a dead pane.
