Integration And Extension Points

The extension surface is distributed across the public app API, MCP tools, git provider abstractions, LLM profile switching, skills/hooks/plugin loading through the agent-server, frontend API conventions, and the enterprise SaaS overlay.

Integration Map

Public App API

V1 API surface

The app-server exposes /api/v1 routes for events, app conversations, pending messages, sandbox management, sandbox specs, settings, secrets, user, skills, webhooks, web client config, git, and config.

Frontend boundary

The frontend path follows the repository guidance: UI components call TanStack Query hooks, hooks use frontend/src/api, and API clients hit app-server endpoints or direct runtime URLs for runtime-specific operations.

Skills, Hooks, And Plugins

SkillsV1 skills and V0 microagents are backed by the same shareable/repository instruction system. Startup asks the sandbox agent-server /api/skills endpoint to load public, user, project, org, and sandbox skills.
HooksThe app-server hook loader explicitly acts as a thin proxy; the agent-server loads hooks from the project directory and returns parsed hook config.
PluginsSDK plugin sources and parameter dictionaries flow into conversation settings and can also append parameter text to the initial message path.

Enterprise Overlay

Enterprise SaaS startup sets OPENHANDS_CONFIG_CLS to a SaaS config class unless already set, disables the base SPA mount while enterprise routes are registered, imports the base app, registers auth, billing, sharing, organization, integration, API key, verified model, email, analytics, and provider-specific routers, adds middleware, then mounts the SPA last.

Architecturally, Enterprise changes app mode, store classes, auth class, analytics user provider, feature flags, and provider configuration. It also introduces SaaS auth with cookie and bearer/API key paths plus effective-org resolution.
