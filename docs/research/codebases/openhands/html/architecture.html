Core Architecture

OpenHands Local GUI is best read as an orchestration application around the OpenHands SDK and sandbox-hosted agent-server. The app-server owns web/API composition, persistence, user/session state, sandbox lifecycle, extension loading, provider integration, and the handoff into the actual running agent process.

Architectural Thesis

Primary boundaryThe app-server builds a fully populated SDK start request and POSTs it to the sandbox agent-server rather than executing the live agent loop in the web process.
Dominant dependency directionRouters depend on injected service interfaces, which resolve configured implementations for storage, sandbox, providers, MCP, settings, and conversation services.
Extension strategyDeployment behavior changes through injectors, store classes, and enterprise dynamic imports more often than through router rewrites.

Backend Composition

App construction

The app initializes the Tavily MCP proxy, builds a FastMCP HTTP app at /mcp, combines its lifespan with the configured app lifespan, creates the FastAPI app, mounts routers, optionally mounts frontend/build, and installs middleware.

Router ownership

The V1 router is a composition layer. It aggregates events, app conversations, pending messages, sandbox management, settings, secrets, user, skills, webhooks, web client config, git, and config routes while feature modules keep their own behavior.

Refs: external/OpenHands/openhands/app_server/v1_router.py:23, external/OpenHands/openhands/app_server/v1_router.py:37.

Dependency Direction

The useful mental model is not "routers call implementation classes directly"; it is "routers ask the configured server graph for a service". This is why the enterprise overlay can replace stores and auth while preserving most route shapes.

Runtime Boundary

Owned by app-serverOwned by sandbox agent-server/runtimeWhy it matters
User, settings, secrets, LLM profiles, provider tokens, metadata, event callback registration.Running conversation endpoint, event ingestion for a live conversation, skills/hooks loading, workspace-local execution.Most durable state and policy lives outside the sandbox; live execution depends on reachable runtime URLs and session keys.
Startup orchestration, sandbox selection, request assembly, persistence, pending message replay.SDK request execution, tool runtime, workspace operations, runtime-specific endpoints such as VSCode and upload.Failures split into orchestration failures versus runtime failures; debugging should first locate which side owns the failing step.
Enterprise auth, organization scoping, SaaS stores, analytics, billing/sharing routers.Same agent-server contract, but with tenant-scoped state fed into the request and callbacks.The enterprise overlay changes policy and storage, not the basic app-server to sandbox agent-server handoff.

Source Layout Signal

openhands/app_server/

Current V1 ASGI app, service interfaces, injectors, routers, conversation lifecycle, stores, sandbox adapters, and integration glue.

openhands/server/

Compatibility wrapper; listen.py re-exports openhands.app_server.app:app.

frontend/

React SPA using React Router, TanStack Query, Axios, Monaco, xterm, Zustand, HeroUI, i18n, and Vite/React Router tooling.

skills/

Shareable instruction/microagent material used by the V1 skills path and legacy V0 terminology.

enterprise/

Source-available SaaS/Enterprise overlay that stacks middleware and replaces implementations through config and imports.
