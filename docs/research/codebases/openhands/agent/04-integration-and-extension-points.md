# Integration And Extension Points

## Public App API

The app-server exposes a V1 REST surface under `/api/v1`, aggregating events, app conversations, pending messages, sandbox management, sandbox specs, settings, secrets, user, skills, webhooks, web client config, git, and config routes (`external/OpenHands/openhands/app_server/v1_router.py:23`, `external/OpenHands/openhands/app_server/v1_router.py:37`). The frontend axios client targets the current host or `VITE_BACKEND_BASE_URL`, then centralizes a response interceptor for selected auth-related 403 handling (`external/OpenHands/frontend/src/api/open-hands-axios.ts:1`, `external/OpenHands/frontend/src/api/open-hands-axios.ts:60`).

Several browser operations intentionally cross from app-server APIs to runtime APIs. The V1 frontend client builds URLs from `conversation_url` and sends session API headers for runtime-specific VSCode, pause/resume, ask-agent, upload-file, and runtime conversation calls (`external/OpenHands/frontend/src/api/conversation-service/v1-conversation-service.api.ts:29`, `external/OpenHands/frontend/src/api/conversation-service/v1-conversation-service.api.ts:35`, `external/OpenHands/frontend/src/api/conversation-service/v1-conversation-service.api.ts:165`, `external/OpenHands/frontend/src/api/conversation-service/v1-conversation-service.api.ts:520`).

## MCP

The app-server mounts a FastMCP server at `/mcp` (`external/OpenHands/openhands/app_server/app.py:33`, `external/OpenHands/openhands/app_server/app.py:59`). `mcp_router.py` also initializes an optional Tavily proxy when an API key exists, so the key stays server-side (`external/OpenHands/openhands/app_server/mcp/mcp_router.py:43`, `external/OpenHands/openhands/app_server/mcp/mcp_router.py:76`).

MCP tools are used for provider-side pull/merge request creation. The router defines tools such as `create_pr`, `create_mr`, Bitbucket PR, Bitbucket Data Center PR, and Azure DevOps PR, resolving provider tokens/user context and updating conversation metadata when PR numbers can be extracted from tool output (`external/OpenHands/openhands/app_server/mcp/mcp_router.py:98`, `external/OpenHands/openhands/app_server/mcp/mcp_router.py:212`, `external/OpenHands/openhands/app_server/mcp/mcp_router.py:215`, `external/OpenHands/openhands/app_server/mcp/mcp_router.py:286`, `external/OpenHands/openhands/app_server/mcp/mcp_router.py:289`, `external/OpenHands/openhands/app_server/mcp/mcp_router.py:421`).

## Git Providers

The provider layer defines `ProviderType` values for GitHub, GitLab, Bitbucket, Bitbucket Data Center, Forgejo, Azure DevOps, and Enterprise SSO (`external/OpenHands/openhands/app_server/integrations/service_types.py:16`, `external/OpenHands/openhands/app_server/integrations/service_types.py:24`). It also defines the `GitService` protocol for repository, branch, suggested-task, PR, and user operations (`external/OpenHands/openhands/app_server/integrations/service_types.py:132`, `external/OpenHands/openhands/app_server/integrations/service_types.py:330`).

`ProviderHandler` maps provider types to service classes, aggregates repo/task/branch results, verifies repository-provider compatibility, and builds authenticated git URLs (`external/OpenHands/openhands/app_server/integrations/provider.py:105`, `external/OpenHands/openhands/app_server/integrations/provider.py:165`, `external/OpenHands/openhands/app_server/integrations/provider.py:410`, `external/OpenHands/openhands/app_server/integrations/provider.py:663`). The `/git` router exposes provider installations, repositories, branches, search, and suggested task endpoints using that handler (`external/OpenHands/openhands/app_server/git/git_router.py:49`, `external/OpenHands/openhands/app_server/git/git_router.py:331`).

During startup, selected repositories are cloned in the sandbox using authenticated URLs from the user context. Azure DevOps has a special bearer-token clone path (`external/OpenHands/openhands/app_server/app_conversation/app_conversation_service_base.py:352`, `external/OpenHands/openhands/app_server/app_conversation/app_conversation_service_base.py:405`).

## LLM Profiles And Model Switching

Settings hold `agent_settings`, `conversation_settings`, and `llm_profiles`; the frontend and backend treat profiles as named, switchable LLM configurations (`external/OpenHands/openhands/app_server/settings/settings_models.py:114`, `external/OpenHands/openhands/app_server/settings/settings_models.py:155`). During conversation startup, the service configures LLM and MCP for the SDK request and, when enough valid profiles exist, attaches the built-in `SwitchLLMTool` to the agent (`external/OpenHands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:1404`, `external/OpenHands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:1482`).

Profiles are also mirrored into the sandbox agent-server because the agent's switch tool resolves from sandbox storage while SaaS profiles live on app-server/org records (`external/OpenHands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:786`, `external/OpenHands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:862`). The frontend exposes profile switching through `/api/v1/app-conversations/{id}/switch_profile` and ACP model switching through `/switch_acp_model` (`external/OpenHands/frontend/src/api/conversation-service/v1-conversation-service.api.ts:237`, `external/OpenHands/frontend/src/api/conversation-service/v1-conversation-service.api.ts:261`).

## Skills, Hooks, And Plugins

The `skills/README.md` explains that V1 "skills" and V0 "microagents" are backed by the same shareable/repository instruction system, with shareable skills in `OpenHands/skills/` and repository-specific instructions in `.openhands/skills/` or `.openhands/microagents/` (`external/OpenHands/skills/README.md:1`, `external/OpenHands/skills/README.md:51`, `external/OpenHands/skills/README.md:69`).

For V1 startup, the app-server does not parse every skill source itself. `load_and_merge_all_skills()` calls the sandbox agent-server `/api/skills` endpoint and asks it to load public, user, project, organization, and sandbox skills (`external/OpenHands/openhands/app_server/app_conversation/app_conversation_service_base.py:97`, `external/OpenHands/openhands/app_server/app_conversation/app_conversation_service_base.py:147`). The app-server then merges returned skills into the agent context (`external/OpenHands/openhands/app_server/app_conversation/app_conversation_service_base.py:161`, `external/OpenHands/openhands/app_server/app_conversation/app_conversation_service_base.py:180`).

Hooks are similarly delegated. `hook_loader.py` states that the app-server is a thin proxy and all hook loading is handled by the agent-server; it POSTs the project directory to `/api/hooks` and returns a parsed `HookConfig` if non-empty (`external/OpenHands/openhands/app_server/app_conversation/hook_loader.py:1`, `external/OpenHands/openhands/app_server/app_conversation/hook_loader.py:80`, `external/OpenHands/openhands/app_server/app_conversation/hook_loader.py:126`). Conversation startup treats hook load failures as non-blocking (`external/OpenHands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:1487`, `external/OpenHands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:1506`).

Plugins are SDK `PluginSource` values extended with parameter dictionaries for user-provided configuration. Startup passes plugin sources into conversation settings and also appends plugin parameter text to the initial message path (`external/OpenHands/openhands/app_server/app_conversation/app_conversation_models.py:62`, `external/OpenHands/openhands/app_server/app_conversation/app_conversation_models.py:72`, `external/OpenHands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:1507`, `external/OpenHands/openhands/app_server/app_conversation/live_status_app_conversation_service.py:1532`).

## Frontend Integration Boundary

The repository guidance states the frontend data path as: UI components -> TanStack Query hooks -> `frontend/src/api` -> API endpoints (`external/OpenHands/AGENTS.md:148`, `external/OpenHands/AGENTS.md:154`). The code follows this shape in the sampled paths:

- Routes define login, onboarding, home, launch, settings subsections, conversation, OAuth device verify, and shared conversation pages (`external/OpenHands/frontend/src/routes.ts:8`, `external/OpenHands/frontend/src/routes.ts:49`).
- Query client config centralizes mutation retry for 429, query error toast behavior, and 401 invalidation for auth state (`external/OpenHands/frontend/src/query-client-config.ts:1`, `external/OpenHands/frontend/src/query-client-config.ts:63`).
- Query hooks wrap service calls, for example `useUserConversation()` batch-fetches V1 app conversations and skips start-task placeholders (`external/OpenHands/frontend/src/hooks/query/use-user-conversation.ts:23`, `external/OpenHands/frontend/src/hooks/query/use-user-conversation.ts:44`).
- Skills and hooks have dedicated query hooks backed by V1 conversation service endpoints (`external/OpenHands/frontend/src/hooks/query/use-conversation-skills.ts:6`, `external/OpenHands/frontend/src/hooks/query/use-conversation-hooks.ts:36`).

## Enterprise Overlay

Enterprise SaaS startup sets `OPENHANDS_CONFIG_CLS` to `server.config.SaaSServerConfig` unless already set, disables the base SPA mount until enterprise routes are registered, imports the base app, registers auth, billing, sharing, organization, integration, API key, verified model, email, analytics, and provider-specific routers, adds enterprise middleware, then mounts the SPA last (`external/OpenHands/enterprise/saas_server.py:7`, `external/OpenHands/enterprise/saas_server.py:206`).

`SaaSServerConfig` changes app mode, store classes, auth class, analytics user provider, feature flags, and provider configuration exposed to the frontend (`external/OpenHands/enterprise/server/config.py:55`, `external/OpenHands/enterprise/server/config.py:188`). `SaasUserAuth` supports cookie and bearer/API key auth, resolves an effective org id with precedence rules, scopes settings/secrets/API keys to that org, and exposes organization role/permissions (`external/OpenHands/enterprise/server/auth/saas_user_auth.py:56`, `external/OpenHands/enterprise/server/auth/saas_user_auth.py:274`, `external/OpenHands/enterprise/server/auth/saas_user_auth.py:337`, `external/OpenHands/enterprise/server/auth/saas_user_auth.py:606`).

Enterprise architecture docs describe SaaS authentication as Keycloak/OIDC-based with signed JWT cookies and provider tokens stored separately for git operations (`external/OpenHands/enterprise/doc/architecture/authentication.md:1`, `external/OpenHands/enterprise/doc/architecture/authentication.md:58`). The README warns enterprise is brittle around user-id versus token assumptions because OSS historically revolves around a user-provided GitHub token while Enterprise revolves around a GitHub/Keycloak user id (`external/OpenHands/enterprise/README.md:49`, `external/OpenHands/enterprise/README.md:56`).
