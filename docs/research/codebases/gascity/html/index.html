02

Core Architecture Thesis

What Gascity Is

Gas City is a configurable orchestration runtime for multi-agent coding workflows. It is not a fixed role application. The important domain objects live under internal/*, and the CLI, HTTP/SSE API, and dashboard project those objects outward.

The most useful mental model is a bead-backed desired-state engine: config and work demand describe what sessions should exist; CityRuntime repeatedly reconciles that desired state to live runtime providers; Beads hold durable work and session lifecycle; Events record infrastructure observations.

What Is Central

  • cmd/gc/cmd_supervisor.go: machine supervisor startup, registry, city runtime construction.
  • cmd/gc/city_runtime.go: per-city controller loop, reload, orders, desired state, cleanup, shutdown.
  • cmd/gc/build_desired_state.go: desired session state, demand, partial-query flags, scale checks.
  • cmd/gc/session_reconciler.go: bead-driven wake, sleep, drain, restart, and create decisions.
  • internal/{config,beads,session,worker,runtime,events,formula,molecule,sling,orders}: domain model and extension seams.

04

How Orchestration Actually Runs

Stage What Happens Load-Bearing Files Why It Matters
1. Config load city.toml is loaded with includes, pack expansion, provider catalogs, agents, named sessions, rigs, formulas, orders, services, API, convergence, and validation. internal/config/config.go, compose.go, pack.go, resolve.go Config is activation. A pack/include change can alter watch targets, agent definitions, order layers, provider selection, and runtime reload behavior.
2. Start request The normal gc start path resolves the city, runs pre-start checks, then registers it with the machine supervisor. Foreground/controller mode is a compatibility path. cmd/gc/cmd_start.go Do not assume the CLI owns the long-running controller. The supervisor usually owns the durable runtime loop.
3. Supervisor reconcile The supervisor holds a registry, API listener, control socket, event recorder, and live city map. It applies backoff, loads city config, prepares dependencies, opens events/stores, creates a provider, and constructs a CityRuntime. cmd/gc/cmd_supervisor.go, internal/supervisor/*, internal/api/supervisor.go This is the boundary between machine-level process ownership and per-city orchestration.
4. Runtime startup CityRuntime.run starts config watching, provider tracking, store setup, telemetry, health gates, startup cleanup, initial session sync/reconcile, and convergence startup reconcile. cmd/gc/city_runtime.go The city is not ready just because the process exists. Startup establishes store, provider, event, service, and session-control state.
5. Tick phases Each tick handles dirty config/reload, filesystem pressure, managed-Dolt preflight, due orders, session snapshots, dead-runtime cleanup, demand load, desired state, bead/index sync, reconciliation, wisp GC, services, auto-suspend, and convergence. city_runtime.go, build_desired_state.go, order_dispatch.go This is the main control loop. Order dispatch intentionally happens before expensive session reconciliation so formula-triggered work is not starved.
6. Desired state DesiredStateResult combines config, store reads, session bead snapshots, runtime provider observations, scale checks, demand, assigned-work snapshots, and partial-query markers. cmd/gc/build_desired_state.go Partial store/session flags are safety-critical. Incomplete evidence must defer destructive drains rather than imply missing work.
7. Reconciliation The bead-driven reconciler checks provider health, dependency order, circuit breakers, stale pending-create rollback, undesired sessions, desired liveness, drain acknowledgements, restart requests, and bounded start waves. cmd/gc/session_reconciler.go, session_lifecycle_parallel.go This is where durable intent becomes live process action. It is deliberately guarded because false negatives can kill useful sessions.
8. Shutdown Supervisor shutdown cancels city contexts. City shutdown waits for async starts/stops, closes trace/service state, optionally preserves sessions for re-adoption, drains orders, and gracefully or forcibly stops visible provider sessions. cmd/gc/cmd_supervisor.go, cmd/gc/city_runtime.go Shutdown is not a simple process kill. It decides whether sessions should be preserved, drained, or stopped.

Work Creation

Work usually enters through Beads, formulas, sling, or orders. A formula compiles to a recipe; a molecule writes that recipe into a bead graph; sling routes or attaches intent; an order creates tracking rows before launching formula or exec work.

Session Creation

Sessions are bead-backed. Immediate creation writes active metadata and starts a provider. Deferred creation writes state=start-pending with a claim so the controller can safely start it later.

Observation

Events record infrastructure facts such as session lifecycle, bead operations, orders, controller state, and provider health. Agent messages and tool output live in provider transcript/log surfaces, not just events.

06

Session And Work Mechanics

Worker Boundary

internal/worker.Handle is the current migration boundary for lifecycle, messaging, transcript, interaction, peeking, and live observation. New CLI/API session work should verify whether it is expected to route through worker.Handle or whether a direct session.Manager path is an allowed exception.

Why this matters: direct session-manager construction still appears in some paths, but upstream guidance identifies the worker handle as the production boundary for session lifecycle.

Start Safety

Starts are fenced by durable session metadata, pending-create claims, generation, continuation epoch, instance token, startup timeout, stale key detection, provider-health snapshots, and bounded parallel start waves.

This is why a simple "start if not running" rewrite would be dangerous. The controller is protecting against duplicate starts, stale provider state, partial stores, and recovery races.

Formula, Molecule, Sling, And Orders Flow

  1. internal/formula loads a workflow template, resolves inheritance, applies control flow/advice/aspects/conditions/retries/graph controls, and flattens to a recipe.
  2. internal/molecule instantiates the recipe as durable bead graph state, or attaches a sub-DAG with idempotency and expected-epoch fencing.
  3. internal/sling exposes intent-style operations: route a bead, launch a formula, attach a formula, or expand a convoy. It depends on narrow interfaces instead of owning stores directly.
  4. internal/orders maps cooldown, cron, condition, event, or manual triggers to formula or exec dispatch. Runtime dispatch creates tracking beads before launching work.
07

Integration And Extension Surfaces

CLI

The gc command tree covers city lifecycle, supervisor, sessions, sling, orders, formulas, convergence, beads, events, hooks, config, packs, services, and diagnostics. Some mutations are direct local operations; others route through the supervisor for lock coordination.

HTTP/SSE API

The API is a typed Huma projection over supervisor and per-city routes. OpenAPI is generated from live route registration. The dashboard is a static SPA over that API, not a control-plane owner.

Runtime Providers

runtime.Provider is the live-session contract: start, stop, interrupt, attach, liveness, nudge, metadata, peek, list, activity, copy, raw keys, live reconfiguration, and capabilities.

Bead Stores

[beads] can resolve to file, exec, native Dolt, or bd CLI-backed paths. Preflight and fallback decisions mean tests should pin provider mode intentionally.

Config/Packs

Agents, named sessions, rigs, providers, packs, formulas, orders, hooks, services, GitHub settings, convergence, and runtime-specific fields are declarative extension points.

Environment

Important controls include GC_SESSION, GC_EVENTS, GC_BEADS, GC_HOME, XDG_RUNTIME_DIR, GC_ISOLATED, and diagnostic toggles such as supervisor pprof.

Risks And Review Rules

Risk Severity Mitigation
Partial store/session reads are mistaken for complete evidence and live sessions are drained or closed. high Preserve StoreQueryPartial, SessionQueryPartial, assigned-work snapshots, and provider-health gates. Add characterization tests before simplifying.
Session lifecycle changes bypass worker.Handle or duplicate worker/session-manager responsibilities. medium Before editing session CLI/API paths, trace whether the production contract is worker-backed or an explicit direct-manager exception.
Configured provider names are treated as concrete implementations. medium Inspect factory/preflight/fallback paths. Pin provider mode in tests instead of assuming bd, Dolt, file, or exec behavior from config alone.
Events are used as authoritative agent transcripts. medium Use events for infrastructure facts. Use provider transcript/log discovery for conversation, tool call, and model-output facts.
Config-only changes accidentally alter reload/watch/runtime behavior. medium Check include/pack provenance, order scans, provider resolution, soft reload acceptance, and runtime watcher behavior.
Parent repo edits accidentally modify submodule internals. high For parent-level research, edit only docs/research/codebases/gascity/. Treat external/gascity as source material unless the task is explicitly submodule-local.
