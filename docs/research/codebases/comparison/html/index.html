01

Executive Summary

The core difference is the level where each project chooses to own orchestration. Gastown owns local multi-agent work through towns, rigs, Beads, git worktrees, tmux sessions, and a merge queue. Gascity owns a configurable desired-state runtime that reconciles city.toml, packs, formulas, orders, beads, and runtime providers. Paperclip owns a server-side product control plane with Postgres-backed work, governance, queues, environments, plugins, and adapters. OpenHands owns the app-server and frontend around sandboxed software-engineering conversations, while delegating actual agent execution to a sandbox-hosted agent-server.

Gastown

Durable local agent town

Solves durable coordination for many coding agents across repos. It treats git worktrees, Beads/Dolt records, tmux sessions, mail/events, and merge queue state as one operational system.

Solves configurable multi-agent workflows without hardcoded roles. It turns config, demand, stores, providers, formulas, and orders into reconciled live sessions.

Paperclip

Solves productized agent operations: teams, issues, budgets, workspaces, plugins, governance, adapters, live events, and recovery around a central server.

OpenHands

Sandboxed conversation app

Solves user-facing software-engineering conversations. The app-server manages metadata, settings, sandboxes, repositories, skills, hooks, and provider integration, then starts the real conversation inside the sandbox agent-server.

02

Solution Space Map

Gastown and Gascity are closest to orchestration infrastructure. Paperclip and OpenHands are closer to end-user products, but they differ sharply: Paperclip centralizes execution policy in its own server, while OpenHands keeps the web app as an orchestration layer around a separate sandbox agent-server.

04

Runtime Flows

Gastown gt sling validates a bead, allocates or reuses a polecat, creates a worktree and assignment hook, starts a tmux agent session, then gt done or gt mq submit moves completed work into a refinery merge queue.
Gascity gc start registers a city with the supervisor. The supervisor builds a CityRuntime, watches config, loads demand, builds desired session state, and runs a bead-driven reconciler against runtime providers.
Paperclip A user, timer, routine, dependency release, or API call enqueues a wakeup. The heartbeat service claims a queued run, checks policy and locks, creates context, realizes workspace/environment, calls an adapter, then persists outcome and cost.
OpenHands The frontend creates a conversation and receives a start task. The app-server waits for a sandbox, prepares repository/setup/skills/hooks, posts a start request to the sandbox agent-server, stores metadata, and forwards later messages or runtime calls with guards.
Key distinction

Who owns live execution?

Gastown and Gascity own live session lifecycles directly through local process providers such as tmux. Paperclip owns the run orchestration and calls adapters, but environment drivers and adapter packages execute the runtime-specific work. OpenHands intentionally steps back after conversation startup: the sandbox agent-server owns the running conversation.

Key distinction

How work becomes durable

Gastown and Gascity make Beads central. Paperclip uses a relational product schema where issues, wakeups, runs, workspaces, costs, plugins, and runtime services are all durable rows. OpenHands splits metadata/start tasks into SQL, event bodies into file/object storage, and live sandbox facts into sandbox services.

Integration And Extension Seams

Gastown

Agent presets and runtime-distributed content

Agent support is centralized in preset metadata: commands, resume behavior, process names, hooks, instruction files, config dirs, and ACP support. Hooks and plugin content are installed or synced into runtime directories rather than linked as Go packages.

Gascity

Config, providers, packs, formulas, orders

Extension primarily happens through config and interfaces: runtime providers, bead store providers, packs, rich agent definitions, formulas/molecules, and order triggers. The HTTP/SSE API is a typed projection over the same object model.

Paperclip

Adapters and out-of-process plugins

Adapters execute concrete AI runtimes. Plugins are separate workers over JSON-RPC stdio and can contribute jobs, tools, webhooks, UI, migrations, and external adapters. CLI, UI, and MCP are intentionally API clients.

OpenHands

Injected services and sandbox delegation

Deployment behavior changes through app-server injectors and enterprise overrides. Skills, hooks, and plugin sources are included in the conversation start request, but loading and execution are delegated to the sandbox agent-server.

Tradeoffs And Reuse Implications

Project Strongest architectural idea to reuse Primary tradeoff Watch before copying
Gastown Local git/worktree plus Beads coordination, with merge queue and named agent identities as first-class state. Power comes from coordinating several external state systems, not from a single database. Dolt/Beads version assumptions, tmux lifecycle edge cases, provider hook drift, formula/wisp/deacon paths not fully traced.
Gascity Declarative desired-state reconciliation with explicit partial-read safeguards and provider/store abstractions. Flexibility makes config, reload, provider selection, and session reconciliation highly coupled. Worker-boundary migration, provider-specific guarantees, config provenance, event logs not being transcripts.
Paperclip Central product control plane where issues, queues, environments, adapters, plugins, budgets, costs, and governance share one durable model. The service layer is broad; core execution crosses many domains in one heartbeat path. Plugin security, adapter trust, migration history, workspace-finalize barriers, local-trusted threat model.
OpenHands Clean app-server versus sandbox-agent-server boundary, with start tasks hiding slow setup from the initial HTTP response. Critical execution semantics live behind pinned SDK/agent-server packages and runtime APIs. OSS versus Enterprise tenant assumptions, direct frontend runtime URLs, remote sandbox API contract, pinned package internals.
Practical synthesis: for an orchestrators project, Gastown is the richest source for local multi-agent git operations, Gascity is the strongest source for configurable desired-state control, Paperclip is the strongest source for a product-grade server control plane, and OpenHands is the strongest source for sandboxed conversation startup and deployment overlays.

Important open questions carried forward
  • The live tree contains four external repositories, while the current project overlay still documents only gascity and gastown as submodules. Parent-repo tracking status for external/paperclip and external/OpenHands should be clarified before metadata cleanup.
  • Paperclip and OpenHands findings are source-grounded but not runtime-verified; dependency installation, migrations, local boot, UI/API flows, and test suites were not run in the underlying research passes.
  • OpenHands execution semantics behind the pinned openhands-sdk, openhands-agent-server, and openhands-tools packages remain outside the local repository analysis.
  • Gascity provider-specific behavior and the active worker-handle migration need targeted reads before changing session lifecycle paths.
  • Gastown formula/molecule/wisp semantics, gt prime generation, and full Deacon patrol behavior need deeper tracing before relying on those areas as stable interfaces.
