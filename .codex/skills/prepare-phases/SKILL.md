# Prepare Phases

Take one or more input documents and stand up a **workstream**: a brainstorm, a roadmap, and the bd epics and stages that `$run-phases` and `$phase-execution` drive off. Work-state lives in **Beads**; human-readable tracking files are bd-generated and never hand-authored. Model: `.beads/beads.md` -> *Workstream Mirrors*.

## Inputs

- One or more document paths: research notes, design docs, review docs, or spec updates.
- If no paths are provided, ask the user which documents to process.

## Workflow

### Step 1 - Understand The Input

1. Read each input document.
2. Read `AGENTS.md`, `.codex/project/brief.md`, and `.codex/project/docs-index.md`.
3. If `docs/workstreams/status.md` exists, read it as the generated board. If it does not exist yet, use `bd stats`, `bd list`, and `docs/workstreams/README.md` to understand current state.
4. Classify the input:
   - **Research notes**: decisions or tradeoffs that may become implementation work.
   - **New design docs**: new capabilities or components to build.
   - **Review/audit docs**: findings to remediate against existing implementation.
   - **Spec updates**: changes to existing design docs that require implementation updates.

### Step 2 - Brainstorm

Invoke the `$brainstorming` skill with the input documents as context:

1. Determine what needs to happen to integrate this input into the project.
2. Clarify scope, success criteria, and non-goals with the user.
3. Identify phases, deliverables, risk levels, dependencies, and execution order.
4. If real choices exist, present options.
5. Get independent criticism on the brainstorm for standard or deep work when capacity permits.
6. Ensure `docs/brainstorms/` exists, then produce `docs/brainstorms/YYYY-MM-DD-<topic>-requirements.md`. If a brainstorm lifecycle README does not exist yet, do not invent lifecycle moves; keep the requirements file in place and report it.

### Step 3 - Create The Workstream

Ask the user: **new workstream**, or **extend an existing one**?

For a new workstream, create `docs/workstreams/<name>/`:

- `README.md`: charter.
- `roadmap.md`: phased plan and the bd `--spec-id` anchor.
- `plans/`: empty; per-phase plans land here in `$phase-execution`.
- `tracking/`: empty; bd-generated by the renderer.

For flat operational work, the `README.md` charter is the anchor.

### Step 4 - Seed Beads

For each phase:

1. Create the phase epic with `bd create -t epic "<phase>" --spec-id docs/workstreams/<name>/roadmap.md --design docs/brainstorms/<brainstorm>.md --actor "codex:<purpose>" -q`.
2. Create flat stage children with `bd create "<stage>" --parent <epic> -t task --actor "codex:<purpose>" -q`.
3. Declare only real dependencies with `bd dep add <stage> <prereq> --actor "codex:<purpose>"`.
4. If `scripts/bd-render-tracking.sh` exists, run `BD_RENDER=1 bash scripts/bd-render-tracking.sh <name>`. If it does not exist, skip rendering, report the missing renderer, and do not create generated mirror files by hand.

`--actor` is mandatory on every bd write. Conservative git: do not commit unless the active workflow or user grants that authority.

### Step 5 - Summary

Present the workstream path, phase/stage counts, execution order, risk per phase, and next command:

```text
$phase-execution <phase> --roadmap docs/workstreams/<name>/roadmap.md
```

Use `$run-phases docs/workstreams/<name>/roadmap.md` only after the renderer/workstream mirrors are available and the user wants autonomous sequential execution.

## Rules

- Always brainstorm before seeding.
- Always ask the user about new versus existing workstream.
- Every epic belongs to a workstream and carries `--spec-id` plus `--design`.
- Dependencies mirror reality; do not chain by listing order.
- Use repo-relative paths only.
- If input documents are unclear or contradictory, stop and ask before proceeding.
